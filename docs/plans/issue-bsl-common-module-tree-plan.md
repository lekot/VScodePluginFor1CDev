# План: дерево общих модулей и граница с BSL (handoff #25, #21, gap §13)

**Трекеры:** [#21](https://github.com/lekot/VScodePluginFor1CDev/issues/21), [#25](https://github.com/lekot/VScodePluginFor1CDev/issues/25).  
**Продуктовая цель (§13 gap):** не вести пользователя через лишние уровни `Ext` → `Module` → `Module.bsl`; «модуль есть модуль».

## Фаза 1 — Корректные пути на диске

- Убедиться, что `filePath` / `parentFilePath` / команды открытия BSL указыва на реальные файлы в выгрузке Designer и EDT.
- Регрессионные тесты на открытие `Module.bsl` из контекстного меню.

## Фаза 2 — Представление в TreeDataProvider

- Виртуальное сжатие ветки `CommonModules/<Name>/Ext/Module/...` в один или два узла с сохранением идентификаторов для операций.
- Согласование с `PropertiesProvider`: выбор «модуля» показывает свойства из XML метаданных общего модуля, а не промежуточных папок.

## Фаза 3 — Команды и клавиатура

- «Открыть BSL» без обязательного раскрытия цепочки Ext (см. README / `package.json` command `1c-metadata-tree.openBslModule`).

## Фаза 4 — Документация и критерии приёмки

- Обновить пользовательский чеклист [user-workflow-without-edt.md](../analytics/user-workflow-without-edt.md) сценарием «открыл общий модуль из дерева».
- Явно зафиксировать в README, что **семантика BSL** остаётся за LSP; CDT не дублирует анализ кода.

## Фаза 9 (связь с README)

В README указано «фаза 9» для завершения handoff #25: к моменту закрытия #25 должны быть выполнены фазы 1–4 выше или зафиксировано намеренное ограничение в gap-документе.
