## 1. Роли, матрица прав и RLS (критерий EDT-killer **A.5**)

**GitHub:** [#28](https://github.com/lekot/VScodePluginFor1CDev/issues/28) (паритет с EDT), [#18](https://github.com/lekot/VScodePluginFor1CDev/issues/18) (RLS / webview, round-trip).

### Краткая спека

- Закрыть разрыв **A.5** из [edt-killer-target.md](../analytics/edt-killer-target.md): редактор прав покрывает **согласованную с EDT** матрицу объектов/прав для целевого релиза; сохранение RLS и `restrictionTemplate` **без потери данных** при round-trip XML и в webview.
- Матрица паритета (#28) — единый источник истины: какие пары «объект × право» обязаны совпадать с EDT для заявленной версии платформы; остальное явно помечается как «вне паритета / backlog».
- #18 — сценарии из gap §5 (и связанных подпунктов по RLS): воспроизводимые шаги + **регрессионные тесты** на сериализацию/десериализацию XML ролей.

### Критерии приёмки (дух EDT-killer)

1. По таблице паритета (#28) для каждой заявленной строки: действие в CDT даёт XML, который платформа/EDT принимают без ручного «дочищания», либо задокументированное исключение с mitigation.
2. Для RLS/#18: набор эталонных фикстур (минимум один сложный сценарий с `restrictionTemplate`) проходит round-trip **CDT → диск → повторное открытие** без диффа, не считая нормализации, зафиксированной в [manifest.md](../manifest.md).
3. В [edt-killer-target.md](../analytics/edt-killer-target.md) §4 строка **A.5** может быть переведена в **OK** только после закрытия обоих треков или явного сужения scope с ссылкой на матрицу.

### Зависимости

- Gap-документ §12 (роли/RLS), при необходимости — уточнение версии `xmlns` в [manifest.md](../manifest.md).
- Доступ к эталонной конфигурации или фикстурам, согласованным с EDT (команда).

### Риски

- **Скрытая семантика платформы:** расхождение CDT↔EDT без формальной матрицы → бесконечные «а в EDT так». Митигация: зафиксировать матрицу до массовых правок UI.
- **Webview и размер состояния:** регресс производительности при больших ролях. Митигация: см. «wow»-backlog в [developer-backlog.md](../developer-backlog.md) (postMessage вместо встраивания JSON), по необходимости отдельный spike.

## 2. Проверка конфигурации из VS Code / задачи (опора на EDT-killer **D.1**)

**GitHub:** [#30](https://github.com/lekot/VScodePluginFor1CDev/issues/30).

**Замена в очереди:** вместо редактора форм (B.1 / #27) — см. блок об исключении выше.

### Краткая спека

- Закрыть практический контур **«одна команда / одна задача»** для прогона проверки конфигурации (`ibcmd` / платформа) из VS Code, задокументированный в [DEVELOPING.md](../../DEVELOPING.md) и связанных материалах (см. уже начатый PR-контур в [developer-backlog.md](../developer-backlog.md)).
- **Следующий шаг после задач:** отчёт о проверке **в UI расширения** (или явно задокументированный путь «task → артефакт → открыть в терминале» без «тихой дыры»), чтобы критерий D.1 можно было честно продвинуть из «Разрыв».

### Критерии приёмки (дух EDT-killer)

1. Разработчик по документации запускает проверку **без EDT** и получает воспроизводимый лог/exit code (CI-дружественно).
2. Есть явное описание в user journey ([user-workflow-without-edt.md](../analytics/user-workflow-without-edt.md) или README), где лежит отчёт и как его интерпретировать.
3. Строка **D.1** в §4 [edt-killer-target.md](../analytics/edt-killer-target.md) обновляется только при выполнении п.1–2 и согласовании формулировки «из CDT **или** из одной команды задачи» с владельцем продукта.

### Зависимости

- Контракт `ibcmd` и среда (см. [design/e2e-container-matrix-ibcmd.md](../design/e2e-container-matrix-ibcmd.md) / планы матрицы).
- Решение по MVP UI: встроенная панель vs достаточность терминала + ссылка на файл отчёта.

### Риски

- **Разнообразие установок платформы:** пути и версии. Митигация: переменные среды и примеры в DEVELOPING.
- **Объём UI:** полноценная интеграция в webview может конкурировать с A/B. Митигация: поэтапно — сначала стабильный task + документированный артефакт.

## 3. Состав подсистем (**B.3**) и макеты / печать (**B.2**)

**GitHub:** нет одного номера на всё; ориентиры — критерии в [edt-killer-target.md](../analytics/edt-killer-target.md), бэклог [developer-backlog.md](../developer-backlog.md) (фильтр по подсистеме в дереве уже есть; не хватает **редактирования состава**).

**Приоритет владельца** ([AIF-40](/AIF/issues/AIF-40#comment-aa2fc814-607e-41a7-a3e3-e77b1e712bde)): в исполнимую очередь FE **в работу в первую очередь берётся B.3** (редактирование состава подсистем). **B.2** не является блокером для старта B.3 и может вестись отдельным потоком (документация offload / MVP).

### Краткая спека (ADR-лайт)

**B.3 Состав подсистем (основной срез спринта по п.3):** редактирование состава подсистемы из дерева или мастера с **валидацией ссылок** (добавление/удаление объектов в подсистеме без поломки `Configuration.xml` и графа зависимостей). Критерии — как в [edt-killer-target.md](../analytics/edt-killer-target.md) §2.B.3.

**B.2 Макеты / печать:** либо базовый просмотр/редактирование в CDT, либо **явный документированный offload** (внешний инструмент, конфигуратор, скрипт) так, чтобы в user journey не было «тихой дыры». Текущее состояние по §4 — разрыв; решение «только offload» допустимо при обновлении [user-workflow-without-edt.md](../analytics/user-workflow-without-edt.md) и README «известные ограничения».

### Критерии приёмки

1. **B.3 (приоритет):** из UI CDT можно изменить состав подсистемы; невалидные ссылки блокируются с понятным сообщением; автотест на регрессию XML подсистем/конфигурации.
2. **B.2:** в документации описан один из путей: встроенный MVP-редактор **или** пошаговый offload + ссылка на инструмент; в чеклисте «день без EDT» есть явный шаг для макетов/печати.
3. Строки **B.3** и **B.2** в §4 `edt-killer-target.md` обновляются согласованно с выбранной стратегией (не объявлять OK при остаточных «тихих» дырах).

### Зависимости

- Дерево и фильтр по подсистеме (уже есть; см. §4 комментарий про фильтр) — доработка до **редактирования состава** (B.3).
- Для макетов (B.2): решение о формате (Designer XML vs EDT layout) в рамках того же workspace — **после или параллельно** B.3 по ресурсу команды.

### Риски

- **Граф зависимостей подсистем (B.3):** циклы и «осиротевшие» ссылки. Митигация: валидация перед сохранением, эталонные фикстуры.
- **Объём B.2:** полноценный визуальный редактор макетов может съесть весь спринт. Митигация: зафиксировать MVP (например, просмотр + правка XML с валидацией) или жёсткий offload + UX-линк; **не откладывать B.3** из‑за B.2.
