# План реализации: иерархия подсистем (ADR 0001)

**Фаза:** Planning  
**Вход:** ADR 0001 (docs/adr/0001-subsystem-hierarchy.md), контракты из него.  
**Цель:** переход от плоского списка подсистем к иерархическому дереву (родитель–потомки, уникальные id, корректный фильтр).

---

## Зависимости и порядок

- Сначала: извлечение данных о родителе/детях и присвоение уникальных id.
- Затем: построение дерева подсистем в парсерах.
- Затем: интеграция с провайдером (загрузка Content по filePath, фильтр по предкам + Content потомков).

---

## 1. Модель и контракты (без изменения TreeNode)

- **Файлы:** только документация/тесты.
- **Задачи:**
  1.1. Зафиксировать в коде или спеках: у узла подсистемы может быть `properties.parentSubsystemRef` (строка или ref) и при построении дерева — уникальный `id` (path-based: `Subsystems.${name}` для корневой, `${parent.id}.${name}` для дочерней).
  1.2. Зафиксировать: при фильтре по подсистеме видимы объекты из Content **этой** подсистемы и из Content **всех её дочерних** подсистем (рекурсивно).

**Критерии готовности:** контракты зафиксированы (комментарии в коде или тест-спеки); нет изменений в типах TreeNode.

---

## 2. xmlChildObjects — извлечение дочерних подсистем (Конфигуратор)

- **Файл:** `src/parsers/xmlChildObjects.ts`
- **Задачи:**
  2.1. Убедиться, что `extractChildSubsystems(childObjects)` используется при разборе XML подсистемы и возвращает массив имён дочерних подсистем (уже реализовано; при необходимости добавить экспорт/использование в Designer).
  2.2. При необходимости добавить извлечение uuid подсистемы из XML (для хранения в `properties` и отладки).

**Критерии готовности:** Designer при разборе каждой подсистемы получает `childSubsystemNames` из ChildObjects; при наличии — uuid в properties.

---

## 3. Парсер EDT — родитель и построение дерева подсистем

- **Файлы:** `src/parsers/edtParser.ts`
- **Задачи:**
  3.1. **Извлечение родителя:** в `extractPropertiesFromMdo` (или отдельная ветка для типа Subsystem) из Subsystem.mdo извлекать свойство родительской подсистемы (ParentSubsystem или аналог по реальному MDO/документации EDT). Сохранять в `properties.parentSubsystemRef` (нормализованно: имя или id для сопоставления).
  3.2. **Специальная обработка типа Subsystems:** при разборе типа Subsystems (в `parseMetadataType` или после получения списка элементов):
     - собрать плоский список узлов подсистем с заполненным `parentSubsystemRef`;
     - вызвать общую функцию построения дерева по родителю (см. п. 5);
     - присвоить path-based id: корневая `Subsystems.${name}`, дочерняя `${parent.id}.${name}`;
     - установить у узла типа Subsystems в `children` только корневые подсистемы; у каждой подсистемы заполнить `children` дочерними подсистемами и выставить `parent`.
  3.3. **Ленивая загрузка:** при `parseTypeContents(configPath, 'Subsystems')` для EDT загружать все подсистемы (обход всех папок в src/Subsystems с чтением Subsystem.mdo), строить дерево один раз, возвращать только корневые. При раскрытии узла подсистемы дочерние подсистемы уже в дереве (не вызывать повторное чтение MDO).

**Критерии готовности:** в дереве EDT у узла Subsystems только корневые в `children`; у каждой подсистемы уникальный id и корректные parent/children; при отсутствии родителя в MDO подсистема считается корневой (с логированием предупреждения при неразрешённой ссылке).

---

## 4. Парсер Designer (Конфигуратор) — обход всех подсистем и построение дерева

- **Файлы:** `src/parsers/designerParser.ts`, при необходимости `src/parsers/xmlChildObjects.ts`
- **Задачи:**
  4.1. **Обнаружение всех подсистем:** для типа Subsystems не ограничиваться прямыми детьми `Subsystems/`. Рекурсивно обойти пути вида `Subsystems/*.xml` и `Subsystems/*/Subsystems/*.xml` (и глубже), чтобы собрать все узлы подсистем с путём к XML в `filePath`.
  4.2. **Данные из каждого XML:** из каждого файла подсистемы извлекать Name, uuid (если есть), ChildObjects → `childSubsystemNames` (через `findChildObjects` + `extractChildSubsystems`), Content (и другие свойства) в `properties`; `filePath` = путь к этому XML.
  4.3. **Корневые подсистемы:** корневые = те, чьё имя ни разу не встречается в списке дочерних ни у одной другой подсистемы. При неоднозначности — по пути к файлу (корневая та, чей XML в `Subsystems/Имя.xml`).
  4.4. **Построение дерева и id:** присваивать path-based id (корневая `Subsystems.${name}`, дочерняя `${parent.id}.${name}`). Сопоставление «родитель → дочерний узел по имени» при повторяющихся именах — по контексту (путь к файлу вложенной подсистемы лежит под каталогом родителя).
  4.5. **Интеграция в parseMetadataType / parseTypeContents:** при typeName === 'Subsystems' вызывать новую логику (сбор всех подсистем → дерево → только корневые в `children` узла типа Subsystems). При полном парсе (parse) и при parseStructureOnly + parseTypeContents для Subsystems результат должен быть иерархическим.

**Критерии готовности:** в дереве Конфигуратора отображаются вложенные подсистемы; у каждого узла подсистемы уникальный id и правильный filePath; корневые — только в `children` узла Subsystems.

---

## 5. Общая функция построения дерева подсистем (по родителю)

- **Файлы:** один общий модуль (например `src/parsers/subsystemTreeBuilder.ts` или функция в существующем парсере/утилитах), использование из EDT и при необходимости из Designer (если Designer будет строить по родителю после вывода родителя из пути).
- **Задачи:**
  5.1. Реализовать функцию вида `buildSubsystemTree(flatNodes: TreeNode[], rootParent: TreeNode): void`, которая:
     - разбивает узлы на корневые (нет `parentSubsystemRef` или ссылка не разрешена) и дочерние;
     - разрешает родителя по имени/id среди flatNodes;
     - присваивает path-based id;
     - заполняет `parent` и `children`, подвешивает корневые к `rootParent`.
  5.2. При не найденном родителе считать узел корневым и логировать предупреждение.

**Критерии готовности:** одна точка логики построения дерева по родителю; повторно используется в EDT и при необходимости в Designer.

---

## 6. TreeDataProvider — загрузка Content и фильтр

- **Файлы:** `src/providers/treeDataProvider.ts`
- **Задачи:**
  6.1. **Загрузка Content подсистемы:** в `loadSubsystemContent` не строить путь к XML по имени подсистемы (`configPath/Subsystems/${subsystemName}.xml`). Использовать `subsystemNode.filePath`, если задан; иначе fallback на старый путь (для обратной совместимости).
  6.2. **Фильтр по подсистеме:** в `nodePassesSubsystemFilter` оставить проверку «узел — предок выбранной подсистемы» (цепочка `parent`). Добавить учёт Content **всех потомков** выбранной подсистемы в дереве подсистем: рекурсивно собрать id дочерних подсистем и при проверке «входит в Content» учитывать Content выбранной подсистемы и всех её потомков.
  6.3. **eagerLoadSubsystemTypes:** при загрузке типов для фильтра учитывать, что Content может быть у выбранной подсистемы и у её потомков (загружать типы для всех соответствующих Content).

**Критерии готовности:** фильтр по вложенной подсистеме показывает объекты из её Content и из Content всех дочерних подсистем; загрузка Content работает по filePath узла.

---

## 7. MetadataParser / loadElementChildren для подсистем

- **Файлы:** `src/parsers/metadataParser.ts`, `src/parsers/designerParser.ts`, `src/parsers/edtParser.ts`
- **Задачи:**
  7.1. Для типа Subsystems при раскрытии узла подсистемы дочерние подсистемы уже в дереве (не загружаются через loadElementChildren). В `loadElementChildren` / `loadChildrenForElement` для подсистемы возвращать только неподсистемные дочерние элементы (Forms, Ext, Attributes, TabularSections, Content и т.д.).
  7.2. Разрешение пути к элементу подсистемы: для подсистемы с id вида `Subsystems.Закупки.Закупки` и filePath `.../Subsystems/Закупки/Subsystems/Закупки.xml` определять директорию элемента как `path.dirname(element.filePath)` (или эквивалент для EDT). В `MetadataParser.loadElementChildren` при typeName === 'Subsystems' передавать в парсер элемент (или filePath), чтобы путь к папке элемента выводился из filePath, а не из `configPath/Subsystems/${elementName}`.

**Критерии готовности:** раскрытие узла подсистемы показывает дочерние подсистемы из уже построенного дерева плюс остальные дочерние элементы, загружаемые по пути, выведенному из filePath узла.

---

## 8. Тестирование

- **Юнит-тесты парсеров:**
  - **xmlChildObjects:** тесты на `extractChildSubsystems` (несколько Subsystem в ChildObjects, один, пусто, нестандартная структура).
  - **EDT:** тест на извлечение `parentSubsystemRef` из Subsystem.mdo (если есть тестовый MDO); тест на построение дерева подсистем (плоский список с parentSubsystemRef → корневые/дочерние и id).
  - **Designer:** тесты на сбор всех подсистем по рекурсивным путям (FormatSamples с вложенными Subsystems); тест на присвоение path-based id и на то, что корневые — только те, чьё имя не в чьих-либо childSubsystemNames.
- **Фильтр по подсистеме:** в `treeDataProvider.test.ts` (или отдельный файл) тесты: узел — выбранная подсистема или её предок проходит; узел из Content выбранной подсистемы проходит; узел из Content дочерней подсистемы проходит (после реализации п. 6.2); узел из другой ветки не проходит.
- **Интеграционные тесты:** открытие конфигурации с иерархией подсистем (EDT и Designer), проверка структуры дерева (корневые/вложенные), фильтр по вложенной подсистеме и проверка видимости узлов. При наличии — использование FormatSamples (например `ut_demo_ForFormat` с Subsystems/Закупки и Subsystems/Закупки/Subsystems/Закупки.xml).

**Критерии готовности:** перечисленные тесты добавлены/обновлены и проходят; регрессий по существующим тестам нет.

## 9. Открытые моменты (уточнить в реализации)

- Точный формат поля родителя в Subsystem.mdo (EDT): имя тега и структура в XML — уточнить по реальному MDO или документации EDT.
- При необходимости хранить uuid в `properties` для отладки или совместимости (ADR допускает).

## Итоговый порядок шагов (для Implementation)

1. **Модель/контракты** — зафиксировать id, parentSubsystemRef, поведение фильтра (п. 1).
2. **xmlChildObjects** — использование extractChildSubsystems в Designer, при необходимости uuid (п. 2).
3. **Общая функция построения дерева** — buildSubsystemTree по родителю (п. 5).
4. **EDT** — извлечение parentSubsystemRef, специальная обработка типа Subsystems, path-based id (п. 3).
5. **Designer** — обход всех подсистем, childSubsystemNames, корневые по спискам детей, дерево и id (п. 4).
6. **TreeDataProvider** — loadSubsystemContent по filePath, фильтр с Content потомков (п. 6).
7. **MetadataParser / loadChildrenForElement** — путь к элементу подсистемы из filePath (п. 7).
8. **Тесты** — юнит (xmlChildObjects, EDT, Designer), фильтр, интеграция (п. 8).

Каждый пункт — ревьюируемый и коммитабельный шаг. После согласования плана — вход для фазы Implementation (Code → Tests → Code Review).
