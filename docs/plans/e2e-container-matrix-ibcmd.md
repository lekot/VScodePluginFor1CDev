# План сценариев матрицы контейнеров

Связанные документы: [design/e2e-container-matrix-ibcmd.md](../design/e2e-container-matrix-ibcmd.md), код `test/matrix/`.

## 1. Два прохода DFS

1. **Первый проход:** цели верхнего уровня — папки типов (`isRootObjectCreateInTypeFolder`), объекты под ними (`isTopLevelObjectCreateTarget`), папка `Forms` под объектом, с исключениями из `IBMATRIX_SKIP_TYPE_FOLDER_IDS` и узлов без `ChildObjects` (см. `ROOT_TAGS_WITHOUT_CHILDOBJECTS`).
2. **Второй проход:** вложенные цели под объектами с именем `Matrix_*` — контейнеры «Реквизиты», «Табличные части», «Формы» (`isNestedMatrixTargetUnderMatrixObject`).

## 2. Операции на цели

Для каждой цели (где применимо): сценарии создания/дублирования/удаления согласно runner матрицы; детали в `containerMatrix.e2e.test` и связанных модулях.

## 3. Сценарий create×2 → delete×1

### 3.1 Смысл

Проверка устойчивости XML и `Configuration.xml` при **двух** последовательных созданиях и **одном** удалении на одной и той же цели-контейнере (или согласованной паре create/delete), без противоречий с ограничениями ibcmd.

### 3.2 Узлы-контейнеры

Функция **`isMatrixTarget`** в `test/matrix/matrixTargetPredicate.ts` определяет, участвует ли узел в матрице как цель первого прохода. К ним относятся:

- папка `Forms` под объектом;
- папки типов верхнего уровня (кроме исключённых и «хрупких для ibcmd»);
- экземпляры корневых объектов под папкой типа;
- контейнеры `Attributes` / `TabularSections` под объектами (где разрешено по правилам `ChildObjects`).

Не цели: корень `Configuration`, экземпляры форм-листьев, исключённые типы из §4 design-документа.

## 4. Критерий успеха с ibcmd

После серии операций на рабочей копии каталога — опционально `runIbcmdOnWorkDir(workDir)` возвращает `executed` с кодом 0; при `skipped` матрица по файлам всё равно считается пройденной по своим assert.
