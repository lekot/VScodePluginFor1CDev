# Test Review: Manifest-driven Gaps

## Цель ревью

Проверить, что тестовая стратегия проекта защищает от сценария:
**тесты зеленые, а в прод-контуре у пользователей проблемы**.

Фокус: реалистичность тестов, prod-like фикстуры, покрытие критичных пользовательских workflow, надежность test-gate.

## Главный вывод

В проекте есть сильные тестовые зоны (парсинг, часть tree provider, ряд регрессионных кейсов), но есть системный риск:
**основной быстрый test-gate не покрывает значимую часть runtime-поведения VS Code расширения**.

Это значит, что при зеленом быстром прогоне можно пропустить регрессии, которые проявятся у пользователя после передачи файлов в платформу.

## Актуальный статус закрытия (на сейчас)

Статус после выполнения `docs/plans/test-review-remediation-plan.md`:

- **Закрыто (`done`)**
  - Обязательный runtime gate: `tests` + `vscode-tests` + `smoke-tests`.
  - Fail-fast preconditions в smoke (нет "тихого" skip по отсутствию workspace/config root).
  - Усиление properties integration (часть сценариев переведена на observable checks).
  - Watcher suite (burst/debounce/conflict) в детерминированном тестовом контуре.
  - Rights/form/multi-root integration сценарии существенно расширены.
  - Fixture matrix и CI-report реально выполненных suite.

- **Частично закрыто (`partial`)**
  - Async discipline (`no-floating-promises`) внедрен не на весь `test/suite/**/*.test.ts`, а на ограниченный набор критичных entrypoints/сценариев.
  - Часть integration-тестов все еще содержит pass-through/условные ветки, где проверка результата может быть недостаточно строгой.
  - Smoke как gate стал строже, но путь form-editor зависит от наличия форм в текущей smoke-фикстуре.

- **Остается gap (`open`)**
  - Симметричное prod-like покрытие **EDT vs Designer** не доведено до целевого уровня (Designer-путь покрыт заметно лучше).

Вывод по состоянию: **not fully closed** — риск значительно снижен, но не обнулен.

## Findings (по приоритету)

### Critical

1. Критичные runtime-сценарии не являются обязательной частью основного gate.
2. Smoke-контур может проходить без проверки ключевого пользовательского пути (из-за skip/зависимости от внешнего workspace).

### High

3. Часть UI/integration тестов проверяет "не упало", но не проверяет наблюдаемый результат.
4. Есть async-сценарии, где действия в тестах выполняются без `await`, что снижает достоверность проверок.
5. Watcher/external-change конфликты недостаточно покрыты поведенческими тестами.

### Medium

6. Для части сценариев много synthetic/monkeypatch веток при недостатке реальных end-to-end проверок.
7. Есть тесты с условной логикой без fail-fast precondition: при изменении фикстур тест может "ничего не проверить".
8. Покрытие Designer заметно сильнее, чем EDT в реалистичном режиме.

## Матрица "покрытие vs реальность"

- **Высокая уверенность**
  - Lazy loading + stale reference rebinding.
  - Search/filter/subsystem interactions (ядро логики tree provider).

- **Средняя уверенность**
  - Multi-root runtime flow.
  - CRUD операций по элементам (особенно в разрезе полного цикла с актуализацией всех связанных файлов).
  - Properties save/cancel semantics в условиях внешних изменений.
  - Rights/form editor end-to-end сценарии.

- **Низкая уверенность**
  - Watcher-triggered reload и конфликтные внешние изменения в реальном timing-контексте.
  - Надежность smoke как обязательного сигнала качества.

## Оценка фикстур (prod-like)

Текущие фикстуры полезны для базовых сценариев, но недостаточны для уверенного прод-уровня.

Недостающие измерения:
- крупные конфигурации (масштаб и глубина дерева);
- более разнообразные malformed XML кейсы;
- extension overlay и конфликтные наложения;
- конкурентные внешние изменения файлов (race/debounce);
- симметричный набор для EDT и Designer;
- Unicode/локализационные edge cases.

## Почему это критично в контексте манифеста

По манифесту проект несет инженерную ответственность за корректность файлов, которые уходят в реальные контуры.

Некачественный test-gate бьет:
- по репутации разработчика, который отдал изменения;
- по репутации инструмента;
- по бизнесу клиента.

Это признано неприемлемым. Следовательно, тестовая стратегия должна быть не "достаточной локально", а "доказательно надежной для прод-рисков".

## Приоритетный план усиления (Top 10)

1. Сделать обязательным двустадийный gate: `node/mocha` + `vscode integration/smoke`.
2. Запретить "тихий skip" smoke-критериев: отсутствие тестового workspace должно падать, а не пропускать проверку.
3. Усилить `properties` integration: проверять результат и side effects, а не только отсутствие исключения.
4. Привести async тесты к корректному `await`-поведению.
5. Добавить полноценный suite для watcher debounce/conflict/burst-сценариев.
6. Добавить end-to-end тесты rights editor: open -> edit -> save -> reopen -> verify.
7. Добавить lifecycle тесты форм (rename/delete/duplicate), не только create.
8. Добавить multi-root e2e кейсы с проверкой отсутствия cross-root leakage.
9. Ввести fixture matrix (small/medium/large + malformed + overlays + external changes).
10. Сделать отчет о реально выполненных suite в CI-артефактах (чтобы видеть, что именно было протестировано).

## Definition of Done для тест-усиления

Изменения в тест-стратегии считаются достаточными, если:
- критичные workflow покрыты обязательным gate;
- невозможен зеленый gate при пропуске production-critical сценариев;
- фикстуры репрезентативны для реальных пользовательских контекстов;
- риски watcher/external-change и редакторов (properties/rights/forms) имеют проверяемые e2e контракты.
