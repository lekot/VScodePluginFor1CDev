# Функциональный разрыв CDT 41 и EDT: что мешает считать CDT полноценной заменой

Дата: 2026-03-23 (редакция: §0 — преимущества CDT; §2–§8 + Issues; §11 — сводная таблица; §11.3 — матрица боль/стоимость; §12 — отложенный engineering-backlog редактора прав / AI-first).  
Продукт: **CDT 41** (Community Development Tools for 1C) — расширение VS Code/Cursor для дерева метаданных и точечного редактирования XML (форматы выгрузки **EDT** и **Конфигуратора**).  
База сравнения: **1C:Enterprise Development Tools (EDT)** как типовая среда разработки конфигураций 1С:Предприятие на платформе 8.x.

Цель документа: зафиксировать **чего нет** у CDT относительно ожиданий «полной замены EDT», чтобы планировать дорожную карту и не смешивать цели «навигация по метаданным» и «IDE уровня платформы». Итоговое **позиционирование** CDT (см. §9) этим текстом не пересматривается — документ лишь уточняет, что часть «разрывов» закрывается **экосистемой VS Code** и внешними скриптами, а не обязана входить в ядро расширения.

Ниже §0 компенсирует перекос «всё в пользу EDT»: это **не** отменяет разрывов по функциональному паритету, но фиксирует, **зачем** CDT держать в стеке рядом с EDT.

---

## 0. Где CDT сильнее, проще или выгоднее EDT (не про паритет функций)

Сравнение нечестное, если мерить только список мастеров платформы: EDT задуман как **полная IDE 1С**. CDT — **надстройка над файлами** в редакторе общего назначения. Преимущества CDT относятся к **рабочему процессу, стеку и доступности**, а не к тому, что в CDT больше встроенных инструментов 1С.

| Область | В чём плюс CDT | Оговорка |
|--------|----------------|----------|
| **Редактор и экосистема** | VS Code / Cursor: быстрый старт, привычный текстовый UX, встроенный Git и diff, терминал, задачи, тысячи расширений (в т.ч. BSL Language Server, темы, GitLens и т.д.). | EDT на другом стеке (Eclipse); кому привычен Eclipse, выигрыш субъективен. |
| **ОЗУ и «вес» IDE** | EDT строится на **Eclipse + JVM** — на практике у многих разработчиков стек заметно **прожорливее по оперативной памяти** и тяжелее по ощущению отзывчивости, чем связка **VS Code (Electron) + CDT** при сопоставимой **кроссплатформенности** (Windows / macOS / Linux). Для сценария «в основном код, дерево метаданных, git» VS Code часто ощущается **пошустрее**. | Electron тоже может раздуваться при десятках расширений и тяжёлых workspace; это **качественная** оценка, не бенчмарк на конкретной КФ. На слабых машинах выигрыш не гарантирован. |
| **Файлы и Git-first** | Прямая работа с выгрузкой Designer / деревом EDT **как с файлами в репозитории**: привычные `git diff`, PR, ветки, code review без обязательной прослойки «проект EDT». | Команды, которые живут целиком внутри EDT и ИБ, могут не ценить. |
| **Прозрачность изменений** | Метаданные — это XML (и сопутствующие файлы) на диске; операции CDT ориентированы на **предсказуемость** без скрытой магии (см. `docs/manifest.md`). | Низкоуровневый XML не всем удобнее визуальных мастеров EDT. |
| **Навигация по структуре** | Дерево с **ленивой загрузкой**, поиск по имени / синониму / комментарию, **regex**, переход по совпадениям, фильтр по **типу** и по **подсистеме** в одной боковой панели Explorer — удобно для «быстро найти объект и открыть XML». | В EDT свой навигатор и поиск; паритета «удобнее именно здесь» без замеров не заявляем. |
| **Два формата конфигурации** | Один инструмент навигатора для **Designer (XML)** и **EDT**-раскладки в одном workspace (см. README) — полезно при смешанных или переходных процессах. | Покрытие и зрелость по типам объектов не равны EDT. |
| **Лицензия и доступность** | Проект **open source** (MIT), установка из **VSIX** с GitHub; нет привязки к лицензии на EDT как продукт 1С. | Поддержка и гарантии — на сообществе, не на вендоре. |
| **Расширяемость CDT** | Доработки на **TypeScript + VS Code API**; форк и PR проще, чем разработка под Eclipse/EDT. | Объём работы «догнать EDT» всё равно огромен. |
| **AI и агенты** | Cursor / Copilot / Roo Code и т.п. работают **в том же окне**, что и дерево метаданных и XML — единый контекст для «спросил — поправил файл». | Качество зависит от модели и промпта; не замена платформенной проверке. |
| **Проверяемость самого инструмента** | Открытый репозиторий, **автотесты**, смок и матрица контейнеров (см. README) — для контрибьюторов и регрессий это явный плюс. | EDT как продукт 1С тестируется вендором по своим процессам. |

**Итог по §0:** CDT выгоден там, где важны **лёгкий редактор, git, файлы, сторонние расширения и открытый код**, а не замена **всех** мастеров и отладки конфигуратора. Разделы §2–§8 по-прежнему описывают **функциональный** разрыв в пользу EDT.

## 1. Кратко: что CDT уже даёт (контекст)

- Дерево метаданных с ленивой загрузкой, поиском, фильтрами по типу и подсистеме.
- Просмотр/редактирование свойств через веб-панель, открытие исходного XML.
- Операции жизненного цикла элементов: создание, дублирование, удаление, переименование (в границах реализованной логики и типов).
- Визуальный редактор типов реквизитов (ограниченный набор ссылочных видов).
- Редактор прав роли (в т.ч. работа с `restrictionTemplate` / RLS в ветке EDT `Ext/Rights.xml` — контекст и баги: [#18](https://github.com/lekot/VScodePluginFor1CDev/issues/18), паритет с EDT: [#28](https://github.com/lekot/VScodePluginFor1CDev/issues/28)).
- Кастомный редактор форм по `Ext/Form.xml` (в README явно отмечен как **нестабильный прототип**).

Этого **недостаточно**, чтобы заменить EDT как **единственную** среду разработки конфигурации для большинства команд.

## 2. Язык и код: BSL

| Область | EDT / экосистема платформы | CDT и экосистема VS Code |
|--------|----------------------------|---------------------------|
| Подсветка и языковая служба BSL | Встроено в EDT | **Не разрыв для CDT:** расширение **BSL Language Server** (и смежные плагины) даёт подсветку, диагностику и навигацию в том же редакторе |
| Подсказки и «помощник по коду» | EDT, контекст метаданных | **Не разрыв ядра CDT:** агенты вроде **Roo Code** и аналоги (в т.ч. с бесплатными моделями) могут подсказывать по тексту и контексту workspace |
| Синтаксический/семантический контроль уровня платформы | Прямая связь с контуром 1С | В составе **одного** расширения CDT этого нет; частично перекрывается BSL LS и внешними проверками |
| Рефакторинг «метаданные + код» как единый продукт | EDT | По-прежнему **нет** внутри CDT как у EDT; возможны точечные сценарии через связку инструментов |

**Вывод:** CDT по-прежнему не заявляет роль **единой IDE для BSL**; фокус — **структура метаданных и XML**. При этом **подсветка и LSP для BSL** и **агентные подсказки** не следует считать блокером «замены EDT» на стороне редактора — это отдельные слои экосистемы.

**Трекер (остаточный разрыв с EDT):** [#25 — §2 платформенная семантика BSL и сквозной рефакторинг «метаданные + код»](https://github.com/lekot/VScodePluginFor1CDev/issues/25). Смежно: [#21](https://github.com/lekot/VScodePluginFor1CDev/issues/21) (BSL-файл общего модуля в дереве).

## 3. Запросы и отчётность

| Область | EDT | CDT |
|--------|-----|-----|
| Конструктор запросов | Есть (в составе EDT / связанных инструментов) | Нет (в README прямо указано) |
| Помощь по языку запросов, проверка текста | Есть | Нет |
| Редактор **системы компоновки данных (СКД)** | Полноценный UI для схемы КД, наборов полей, макетов | Нет |
| Предпросмотр отчёта из среды разработки | Есть (через клиент/отладку) | Нет |

**Разрыв:** без СКД и конструктора запросов CDT не закрывает типичный контур разработчика отчётов.

**Трекер:** [#26 — §3 конструктор запросов, СКД, предпросмотр отчётов](https://github.com/lekot/VScodePluginFor1CDev/issues/26).

## 4. UI: формы, макеты, интерфейсы

| Область | EDT | CDT |
|--------|-----|-----|
| Визуальный конструктор форм | Зрелый, с привязкой к метаданным и проверками | Частично: кастомный редактор `Form.xml`, **прототип**, ограниченная зрелость |
| Табличные документы, макеты печати | Редактирование и предпросмотр в связке с платформой | Нет специализированного редактора макетов (в README — «просмотра макетов» нет) |
| Общие картинки, стили, интерфейсы | Полноценные редакторы / визуализация | В дереве и XML — да; **визуальных** редакторов уровня EDT нет |
| Командный интерфейс, подсистемы на уровне «состава» | Редактирование состава подсистем, связей | Фильтр по подсистеме есть; **редактирование состава подсистем** — нет (в README отмечено как будущее) |

**Разрыв:** визуальная разработка UI и макетов в CDT не дотягивает до EDT.

**Трекер:** [#27 — §4 формы, макеты, стили/картинки, состав подсистем](https://github.com/lekot/VScodePluginFor1CDev/issues/27).

## 5. Роли, безопасность, администрирование

| Область | EDT | CDT |
|--------|-----|-----|
| Редактор прав | Полный контур для типовых сценариев | Есть отдельный редактор; покрытие и UX могут отличаться от EDT |
| RLS / шаблоны ограничений | Интегрировано в привычный UI | Для EDT-ветки — работа с `restrictionTemplate` (см. [#18](https://github.com/lekot/VScodePluginFor1CDev/issues/18) / [#28](https://github.com/lekot/VScodePluginFor1CDev/issues/28)); полное паритетное UX с EDT не заявлено |
| Профили безопасности, ИБ на уровне EDT | Есть в продуктовой линейке | Нет |

**Разрыв:** для строгого паритета с EDT нужна явная матрица «все виды прав + все объекты + все ограничения» и сравнение с платформенной проверкой.

**Схемная / платформенно-осмысленная валидация текста RLS** (`restrictionTemplate` и т.п.) относится к **паритету редактора прав с EDT** ([#28](https://github.com/lekot/VScodePluginFor1CDev/issues/28)) — «что допустимо в XML с точки зрения ожиданий платформы», а не к транспортному сценарию «текст из webview доехал до файла» ([#18](https://github.com/lekot/VScodePluginFor1CDev/issues/18)). В [#18](https://github.com/lekot/VScodePluginFor1CDev/issues/18) комментарий в трекере имеет смысл оставить: закрытие #18 не снимает задачу валидации содержимого; она живёт в матрице [#28](https://github.com/lekot/VScodePluginFor1CDev/issues/28). Подробнее см. **§12**.

**Трекер:** [#28 — §5 паритет редактора прав и RLS с EDT](https://github.com/lekot/VScodePluginFor1CDev/issues/28). Баг сохранения RLS: [#18](https://github.com/lekot/VScodePluginFor1CDev/issues/18).

## 6. Связь с информационной базой и жизненный цикл

| Область | EDT | CDT |
|--------|-----|-----|
| Загрузка / выгрузка конфигурации в ИБ | Встроенные мастера | **В ядре CDT нет** мастера как в EDT; контур **автоматизируется снаружи**: **`ibcmd`**, обвязка на **OneScript**, расширения VS Code для 1С — это **не принципиальный запрет**, а отсутствие единой встроенной кнопки «как в EDT» |
| Сравнение конфигурации с базой, обновление | Типовые мастера EDT | Нет в CDT; те же внешние средства (платформа, скрипты) остаются на стороне процесса |
| Отладка (тонкий/веб-клиент, сервер, фоновые задания) | EDT + платформа | **Открытый разрыв:** нужен отдельный замысел и реализация (или интеграция с существующими средствами отладки 1С), согласованный с рабочим процессом |
| Публикация, сборка дистрибутивов | Часть контура EDT/поставки | Нет в CDT |

**Разрыв:** встроенный **платформенный** контур EDT «ИБ ↔ конфигурация ↔ отладка» в CDT **не воспроизведён**; при этом **загрузка/выгрузка через файлы** может быть закрыта **скриптами и расширениями**, без расширения границ самого CDT 41. **Отладка** остаётся слабым местом относительно EDT.

**Трекер:** [#29 — §6 отладка и контур «ИБ ↔ конфигурация»](https://github.com/lekot/VScodePluginFor1CDev/issues/29).

## 7. Качество, валидация, командная работа

| Область | EDT | CDT |
|--------|-----|-----|
| Проверка конфигурации перед применением | Встроена в связку с платформой | Нет встроенного запуска проверки 1С из расширения — трекер: [#30](https://github.com/lekot/VScodePluginFor1CDev/issues/30) |
| Ссылочная целостность при массовых правках | EDT + платформа | Частично через логику XMLWriter/операций; **не эквивалент** полной проверке платформы |
| Сравнение/слияние метаданных (3-way, EDT-формат) | Специализированные средства | Нет аналога EDT merge; запрос на «умное» сравнение/объединение — [**Issue #12**](https://github.com/lekot/VScodePluginFor1CDev/issues/12) |
| Задачи, ревью, процессы EDT | Зависят от продукта/версии | Нет |

**Разрыв:** инженерный контур «гарантия как у платформы» в CDT **не воспроизведён**; ответственность за `ibcmd`/конфигуратор остаётся на пользователе (см. также smoke/матрицу в README).

**Трекер:** проверка конфигурации — [#30](https://github.com/lekot/VScodePluginFor1CDev/issues/30); merge / сравнение-объединение — [#12](https://github.com/lekot/VScodePluginFor1CDev/issues/12).

## 8. Прочие объекты метаданных и «хвосты»

В README уже перечислены ориентиры: нет **просмотра макетов**, **состава планов обмена** и многого другого. К системным разрывам отнести также:

- **XDTO**, **HTTP/Web-сервисы**, **интеграционные сервисы** — в дереве и XML могут отображаться, но **нет** специализированных мастеров и валидаторов уровня EDT.
- **Внешние источники данных**, **бизнес-процессы и задачи** — поддержка в дереве не равна полному набору редакторов EDT.
- **Локализация, языки** — нет выделенного контура как в EDT.
- **Расширения конфигурации** — **частично закрыто:** свойства и дерево можно вести **как у отдельной конфигурации** в workspace; создание конфигураций/расширений из инструмента, контекстное «добавить расширение» с корня и развитие сценариев «как в нативных инструментах» ведётся в [**Issue #4 — Создавать новые конфигурации и расширения в инструменте**](https://github.com/lekot/VScodePluginFor1CDev/issues/4). Полный жизненный цикл «как в EDT» по-прежнему не заявлен до закрытия этого контура.

**Трекер (прочие типы и мастера):** [#31 — §8 XDTO, сервисы, ВИД, БП, локализация, планы обмена](https://github.com/lekot/VScodePluginFor1CDev/issues/31). Расширения: [#4](https://github.com/lekot/VScodePluginFor1CDev/issues/4). Платформенно валидное создание отдельных типов: [#22](https://github.com/lekot/VScodePluginFor1CDev/issues/22), [#23](https://github.com/lekot/VScodePluginFor1CDev/issues/23).

## 9. Итоговая формулировка разрыва

**CDT 41** сегодня ближе к **навигатору и точечному редактору метаданных в файлах** (плюс прототипы форм и редактор прав), чем к **полной IDE**. Сильные стороны CDT в стеке VS Code/Git/open source перечислены в **§0**; ниже — про **функциональный** паритет с EDT.

Чтобы говорить о **полноценной замене EDT** для типичной команды 1С, как минимум потребовались бы (в совокупности, не обязательно в одном расширении):

1. Зрелая визуальная разработка форм и макетов + СКД + конструктор/помощник запросов.  
2. Глубокая интеграция с платформой: **проверка конфигурации**, **отладка**; загрузка/выгрузка ИБ может оставаться за **`ibcmd`/OneScript/расширениями VS Code**, но **единый** и предсказуемый контур «как у EDT» без внешней сборки всё ещё отсутствует.  
3. **Паритет с EDT по связке «метаданные + BSL»** в одном продукте: подсветка/LSP и агенты снимают часть боли, но **сквозной рефакторинг и проверки уровня платформы** как в EDT — отдельный ориентир.  
4. Паритет по редактированию состава подсистем, обменам, остальным «вторичным» редакторам EDT.  
5. Средства сравнения/слияния и командной разработки на уровне ожиданий от EDT.

До этого CDT разумно позиционировать как **дополнение** к EDT/конфигуратору или как **специализированный** инструмент для git- и файловых workflow, а не как drop-in замену всего EDT.

### 11.3. Матрица «боль / дёшево» для приоритизации (оценка на 2026-03-22)

Субъективные баллы для планирования, не для KPI. Шкалы:

| Шкала | 1 | 10 |
|-------|---|-----|
| **Боль** | разрыв почти не ощущается | острая боль по типовым сценариям |
| **Дёшево** | очень дорого / долго / рискованно | узкий объём / быстрый выигрыш |

**Формула приоритета:** `Приоритет = 2 × Боль + Дёшево` (максимум **30**). Боль учитывается **вдвое сильнее**, чем «дёшевизна», чтобы не выходили наверх узкие мастера при низкой боли относительно паритета прав, merge и платформы.

## 12. Отложенный engineering-backlog (редактор прав) и AI-first ориентиры

Пункты ниже — не «отмена работ», а **фиксация в гэпе** приоритетов и смысла относительно EDT и устойчивости продукта.

| # | Тема | Приоритет / очередь | Смысл |
|---|------|---------------------|--------|
| 1 | **Полная классическая i18n** (каталог строк как в тяжёлых IDE) | Гэп, без паритета «строка в строку» с EDT | В EDT язык среды **настраиваемый**; у CDT другой стек (VS Code) и другой контур UX. Для **ИИ-ориентированного** использования язык и формулировки часто приходят из модели/контекста агента — «власть над языками» в продуктовом смысле смещается к **мультиязычному AI-слою** и точечной согласованности UI, а не к обязательному полному i18n-каталогу в стиле Eclipse в первую очередь. |
| 2 | **Схемы контрактов** (Zod / аналоги) + **трансляции** между версиями XML/namespace | **Wow-first todo** в гэпе | Обновление выгрузки / xmlns у вендора легко **ломает весь продукт**, если разбор и сообщения размазаны по коду. Цель: описывать входящие/исходящие структуры **схемами** и централизованные **адаптации** (маппинг версий); правка в одном месте чинит контур **целиком** — «дело в шляпе», а не охота по репозиторию. |
| 3 | **Схемная валидация RLS** (допустимость XML шаблонов ограничений) | Паритет прав **[#28](https://github.com/lekot/VScodePluginFor1CDev/issues/28)**, не корень **[#18](https://github.com/lekot/VScodePluginFor1CDev/issues/18)** | #18 — доставка текста из UI в файл; **качество и допустимость** вставляемого RLS относится к паритету с EDT и расширению валидатора — см. абзац в **§5** и `docs/plans/cdt-gap-issues-research-handoff.md` §4. В GitHub: короткий комментарий к [#28](https://github.com/lekot/VScodePluginFor1CDev/issues/28) (и при желании к [#18](https://github.com/lekot/VScodePluginFor1CDev/issues/18)) с отсылкой к этому разделу. |
| 4 | **Начальная загрузка webview через postMessage** вместо встраивания большого JSON в HTML | **Wow**, **низкий** приоритет | Улучшение надёжности, лимитов движка и экранирования; не блокер функционального паритета с EDT. |
