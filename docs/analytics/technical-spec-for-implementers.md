# Техническая спецификация для команды реализации (CDT 41)

**Аудит модулей:** [aif-6-capability-inventory.md](./aif-6-capability-inventory.md).  
**Целевое состояние:** [edt-killer-target.md](./edt-killer-target.md).

## 1. Границы ответственности

- **В scope CDT:** дерево метаданных, разрешение путей к XML/BSL в выгрузке, ограниченная запись в XML (операции, редакторы форм/прав/типов), наблюдение за файлами.
- **Вне scope (намеренно):** полноценный BSL language server, полная платформенная семантика — см. README «CDT и BSL/LSP».

Новые фичи должны **не размывать** эту границу без явного ADR и обновления `docs/manifest.md` (если есть).

## 2. Контракты при записи XML

1. **Парсер → модель → writer:** любая UX-операция, меняющая файлы, должна проходить через слой, где уже есть тест на round-trip для затронутых типов (`XMLWriter`, `formXmlWriter`, `rightsXmlEditWriter` и т.д.).
2. **`Configuration.xml`:** изменения имён/UUID/состава объектов согласовать с `configurationXmlUpdater` и существующими тестами `xmlWriter*.test.ts`.
3. **Кодировка и переносы:** сохранять совместимость с тем, как платформа и EDT ожидают выгрузку (см. существующие фикстуры; не менять нормализацию без тестов).

## 3. Webview-редакторы (формы, права, свойства)

- Протокол сообщений VS Code `postMessage` — версионировать при breaking changes (документировать в коде/CHANGELOG).
- Панели должны **не блокировать** основной поток расширения; тяжёлый I/O — async.
- Smoke **не проверяет** DOM webview; для зрелости форм нужны либо unit-тесты сериализатора, либо расширение smoke (см. [edt-killer-target.md](./edt-killer-target.md) уровень B).

## 4. Дерево и производительность

- Новые типы узлов: обновить `MetadataType`, иконки/лейблы, ленивую загрузку в `treeDataProvider`.
- Поиск/фильтры: избегать полного обхода дерева на каждый keystroke без debounce/инкремента (следовать текущим паттернам в провайдере).

## 5. Чеклист перед PR крупной фичи

- [ ] `npm run verify` (или минимум `lint:ci` + релевантные тесты).
- [ ] Новые фикстуры в `test/fixtures/` для затронутых XML-форм.
- [ ] Обновление пользовательской документации (`README` или `docs/analytics/`) для новых команд.
- [ ] Если затрагивается паритет с EDT — ссылка на issue из gap-документа и строка в [cdt-vs-edt-functional-gap.md](./cdt-vs-edt-functional-gap.md) при необходимости.

## 6. Интеграция с внешним контуром

- Загрузка в ИБ, `ibcmd`, проверка конфигурации — по умолчанию **скрипты и задачи**, не обязательно код расширения; если добавляется кнопка «Проверить» — контракт exit code и логов описать в README.
- **Поддержка и issue:** новые UX-команды, влияющие на разбор багов (например **Copy diagnostics summary**), должны быть отражены в пользовательском чеклисте [`user-workflow-without-edt.md`](./user-workflow-without-edt.md) и в шаблоне [`.github/ISSUE_TEMPLATE/bug_report.md`](../../.github/ISSUE_TEMPLATE/bug_report.md), чтобы репорты оставались воспроизводимыми без ручных уточнений.
