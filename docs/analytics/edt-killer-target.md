# Целевое определение «EDT-killer» для CDT 41 и критерии приёмки

**Связанные материалы:** [cdt-vs-edt-functional-gap.md](./cdt-vs-edt-functional-gap.md), [aif-6-capability-inventory.md](./aif-6-capability-inventory.md).

## 1. Термин

**EDT-killer** в контексте этого продукта — не «клон Eclipse EDT», а **согласованный набор возможностей и процессов**, при которых **типичная команда разработки конфигурации 1С может отказаться от установки EDT** для **ежедневной** работы с метаданными и UI, оставив платформу (конфигуратор, `ibcmd`, проверка) там, где это неизбежно по контракту с 1С.

Позиционирование «только файлы + Git» сохраняется: EDT-killer **не обязан** воспроизводить все мастера EDT, если эквивалент закрыт **документированным** контуром (скрипты, CI, внешние расширения) и **критериями приёмки** ниже.

## 2. Уровни зрелости

### Уровень A — «Metadata daily driver»

Команда использует CDT как **основной навигатор** по конфигурации в VS Code/Cursor; EDT открывается редко.

**Критерии приёмки (все обязательны):**

1. **Дерево и поиск:** ленивое дерево, поиск по имени/синониму/комментарию, regex, фильтры по типу и подсистеме — без деградации времени отклика на эталонной средней конфигурации (см. smoke-фикстуры + внутренний бенчмарк команды).
2. **CRUD метаданных:** создание, дублирование, удаление, переименование для заявленных в документации типов — с автотестами на регрессию XML и `Configuration.xml`.
3. **Два формата:** открытие и корректная идентификация **Designer** и **EDT** layout в одном workspace; формат детектируется автоматически.
4. **Свойства и XML:** панель свойств и «Open XML» дают согласованное представление с файлом на диске; watcher подхватывает внешние правки без поломки дерева.
5. **Роли:** редактор прав покрывает **согласованную с EDT матрицу объектов/прав** для целевого релиза (отдельная таблица паритета — задача [#28](https://github.com/lekot/VScodePluginFor1CDev/issues/28)); сохранение RLS/`restrictionTemplate` без потери данных.
6. **Документация:** пользовательский сценарий «день разработчика без EDT» описан в `README` или `docs/analytics/user-workflow-without-edt.md` (чеклист: открыть проект, найти объект, поправить метаданные, закоммитить).

### Уровень B — «UI parity for forms & layout»

Команда **не уходит в EDT** для типовой правки форм и простых макетов.

**Критерии приёмки:**

1. **Редактор форм:** для `Ext/Form.xml` — стабильный round-trip (открыть → изменить → сохранить → платформенная загрузка без ручного дочищания XML) на **наборе эталонных форм** в репозитории; smoke расширяется проверкой **минимум одного** содержательного сценария в webview (или контрактными snapshot-тестами сериализации).
2. **Макеты / печать:** либо базовый просмотр/редактирование в CDT, либо явный **offload** в документированный внешний инструмент — без «тихих» дыр в workflow.
3. **Состав подсистем:** редактирование состава подсистемы из дерева или мастера с валидацией ссылок.

### Уровень C — «Reporting & queries»

Команда разрабатывает отчёты и СКД без EDT.

**Критерии приёмки:**

1. Конструктор запросов или интеграция с утверждённым внешним конструктором (см. [#26](https://github.com/lekot/VScodePluginFor1CDev/issues/26)) с экспортом в текст запроса в метаданные.
2. Редактор СКД с покрытием **минимального** набора операций (схема, наборы, ресурсы) — критерии детализируются отдельным spec по [#26](https://github.com/lekot/VScodePluginFor1CDev/issues/26).

### Уровень D — «Platform confidence»

Снижение зависимости от ручного прогона в конфигураторе.

**Критерии приёмки:**

1. Запуск **проверки конфигурации** из CDT или из одной команды задачи VS Code с отчётом в UI ([#30](https://github.com/lekot/VScodePluginFor1CDev/issues/30)).
2. Документированный контур отладки ([#29](https://github.com/lekot/VScodePluginFor1CDev/issues/29)) — встроенный или партнёрский; критерий: «шаг в отладчик с точки из UX CDT/VS Code» без обязательного открытия EDT.

## 3. Определение «готово к заявлению EDT-killer»

*(Ниже — неизменные условия публичного заявления; актуальный срез выполнения см. §4.)*

Публично называть продукт **EDT-killer** можно только если одновременно:

- закрыты **уровни A и B** по критериям выше;
- для **C** и **D** есть публичная дорожная карта с датами или явное ограничение «не входит в scope» без противоречия маркетингу;
- выполнена независимая **ревизия gap-документа** (нет незакрытых блокирующих разрывов без mitigation).

## 4. Снимок соответствия критериям (актуализация после PR #34)

Ниже — оценка **текущего main** относительно §2 без пересмотра маркетинговых заявлений. [PR #34](https://github.com/lekot/VScodePluginFor1CDev/pull/34) усилил **операционную** зону (диагностическая сводка для багрепортов, см. [diagnostics-summary-field-dictionary.md](./diagnostics-summary-field-dictionary.md)); продуктовые уровни A–D от этого **не** считаются автоматически закрытыми.

| Зона | Критерий (см. §2) | Статус | Комментарий |
|------|-------------------|--------|---------------|
| A | 1. Дерево и поиск | **Частично OK** | Функциональность есть; формальный бенчмарк «эталонной средней конфигурации» остаётся на стороне команды/smoke. |
| A | 2. CRUD метаданных | **OK** | Покрыто автотестами на регрессию XML (как заявлено в документации по типам). |
| A | 3. Designer + EDT | **OK** | `FormatDetector`, два layout в одном workspace. |
| A | 4. Свойства и XML, watcher | **OK** | Панель свойств, Open XML, перезагрузка при внешних правках. |
| A | 5. Роли / матрица EDT | **Разрыв** | Явная зависимость от [#28](https://github.com/lekot/VScodePluginFor1CDev/issues/28); сценарии RLS — [#18](https://github.com/lekot/VScodePluginFor1CDev/issues/18). |
| A | 6. Документация «день без EDT» | **OK** | [user-workflow-without-edt.md](./user-workflow-without-edt.md), ссылки из README. |
| B | 1. Редактор форм | **Разрыв / прототип** | Критерии уровня B не закрыты; ведётся [#27](https://github.com/lekot/VScodePluginFor1CDev/issues/27). |
| B | 2. Макеты / печать | **Разрыв** | Нет встроенного редактора; нужен явный offload в документации процесса (см. README «известные ограничения»). |
| B | 3. Состав подсистем | **Разрыв** | В дереве есть фильтр по подсистеме; **редактирование состава** подсистемы из CDT не заявлено как готовое. |
| C | Отчёты / СКД / запросы | **Вне scope текущего релиза** | Дорожная карта — [#26](https://github.com/lekot/VScodePluginFor1CDev/issues/26). |
| D | Проверка конфигурации из CDT | **Частично OK** | Добавлен VS Code task контур `ibcmd` (check/import) через `scripts/ibcmd-cli.cjs` с артефактом отчёта `.ibcmd-reports/*` и явным описанием в `DEVELOPING.md` и `docs/analytics/user-workflow-without-edt.md` (task → artifact → interpretar exit code / stderr). UI-отчёт в расширении — следующий шаг. |
| D | Контур отладки | **Разрыв** | [#29](https://github.com/lekot/VScodePluginFor1CDev/issues/29). |

**Итог для заявления EDT-killer (§3):** уровни **A и B по таблице выше ещё не закрыты** (блокеры: роли/RLS, формы, макеты/состав подсистем); **C и D** требуют публичной дорожной карты или явного scope-limit, согласованного с владельцем реализации.

## 5. Синхронизация с Founding Engineer

Любое изменение scope уровней B–D согласовывается с владельцем реализации (Founding Engineer): приоритет ветки (формы vs СКД vs platform), ограничение по объёму milestone, зависимости от внешних инструментов (`ibcmd`, LSP, агенты).
