# ADR 0001: Иерархия подсистем метаданных

**Статус:** Принят (Design)  
**Дата:** 2025-03-17  
**Контекст:** Фаза Design — правильная иерархия подсистем вместо плоского списка.

---

## Контекст и проблема

Сейчас в дереве метаданных подсистемы отображаются **плоским списком**: узел типа «Subsystems» содержит в качестве прямых потомков все подсистемы конфигурации, независимо от того, что в 1С подсистемы образуют **дерево** (подсистема может иметь родительскую подсистему и дочерние).

- **EDT:** в `Subsystem.mdo` есть связь с родителем (в API EDT: `getParentSubsystem()`, дочерние — `getSubsystems()`). На диске структура плоская: `src/Subsystems/<ИмяПодсистемы>/Subsystem.mdo`.
- **Конфигуратор (cf/xml):** иерархия подсистем также задаётся (родитель/дочерние), а не плоским списком.

Из-за плоской модели:
- не совпадает с моделью 1С и ожиданиями пользователя;
- фильтр «по подсистеме» не отражает вложенность (родительская подсистема должна включать все вложенные);
- дерево не позволяет увидеть структуру «подсистема → подподсистемы».

Нужно задизайнить и зафиксировать **иерархическую модель подсистем** и способ её получения из обоих форматов (EDT и Конфигуратор).

---

## Решение

### 1. Модель данных (дерево)

- **Узел подсистемы** (`TreeNode`, `type === MetadataType.Subsystem`) остаётся как есть.
- **Связь родитель–потомок:** задаётся через уже существующие поля `TreeNode.parent` и `TreeNode.children`. Дополнительно для подсистем храним ссылку на родителя в свойствах (для парсеров и восстановления дерева):
  - `properties.parentSubsystemRef` — строка или объект ссылки на родительскую подсистему (как в источнике: имя, uuid или ref). Опционально; отсутствует или пусто — подсистема корневая.

Иерархия в дереве:

- Узел типа **Subsystems** имеет в `children` **только корневые подсистемы** (у которых нет родителя или родитель не указан).
- Каждая подсистема в `children` содержит **дочерние подсистемы** (и при необходимости другие дочерние элементы: Content, формы и т.д.) в своём `children`.
- Объекты из **Content** подсистемы (справочники, документы и т.д.) не становятся прямыми потомками узла подсистемы в дереве типов — они по-прежнему живут под узлами своих типов (Catalogs, Documents, …). Фильтр по подсистеме продолжает определять видимость по полю Content (и по пути в дереве для самих подсистем).

То есть меняется только **способ построения узлов типа Subsystem** и их вложенности друг в друга; остальная структура дерева (типы → объекты метаданных) без изменений.

**Уникальность узла подсистемы (имена могут повторяться):**

- В 1С и в выгрузке **имя подсистемы не уникально**: одна и та же строка (например «Закупки») может быть и родительской подсистемой (со своим набором ChildObjects и Content), и вложенной дочерней подсистемой с **другим** Content. Пример выгрузки конфигуратора: `Subsystems/Закупки.xml` (корневая «Закупки» со списком дочерних) и `Subsystems/Закупки/Subsystems/Закупки.xml` (вложенная «Закупки» со своим Content).
- В дереве и в фильтре должен быть однозначный ответ на вопрос «какая из них»: не по имени, а по **уникальному идентификатору узла**.
- **Требование:** у каждого узла подсистемы `id` должен быть **уникален в рамках конфигурации**. Варианты:
  - **По пути в дереве:** `id = цепочка имён от корня`, например `Subsystems.Закупки` для корневой и `Subsystems.Закупки.Закупки` для вложенной. При построении дерева: для корневой `id = Subsystems.${name}`; для дочерней `id = ${parent.id}.${name}`. Два узла с одним отображаемым именем получают разные id.
  - **По uuid:** в XML конфигуратора у `<Subsystem>` есть атрибут `uuid`; в EDT он тоже есть. Можно хранить uuid в `properties` и задавать `id = Subsystems.${uuid}`. Однозначно и стабильно при переименовании.
- **Рекомендация:** path-based id при построении дерева (проще сопоставление «родитель по имени» при сборке дерева по childSubsystemNames; uuid при необходимости хранить в properties для отладки или совместимости). В любом случае фильтр и дерево опираются на **id** выбранного узла и его `filePath` (откуда грузится Content), так что «какая подсистема» определяется выбранным узлом, а не строкой имени.

### 2. Построение дерева подсистем

**Общий алгоритм (и для EDT, и для Конфигуратора при наличии данных о родителе):**

1. Получить все подсистемы в виде плоского набора узлов (как сейчас), с заполненными свойствами из MDO/XML, включая ссылку на родителя (см. ниже).
2. По ссылке на родителя разбить подсистемы на:
   - корневые: нет `parentSubsystemRef` или ссылка пустая/неразрешённая;
   - дочерние: есть `parentSubsystemRef`.
3. Построить дерево:
   - корневые подсистемы — прямые `children` узла типа «Subsystems»;
   - для каждой подсистемы в `children` положить только те подсистемы, у которых `parentSubsystemRef` указывает на неё (по имени или id), рекурсивно.
4. Установить `parent` и `children` у каждого узла подсистемы согласно этому дереву.

Разрешение родителя: по строке вида `Subsystem.ИмяРодителя` или по uuid — сопоставление с уже загруженными подсистемами по имени или id. Если родитель не найден (битая ссылка, выгрузка неполная), подсистему считать корневой и логировать предупреждение.

### 3. Источники данных о родителе

**EDT (Subsystem.mdo):**

- В Properties может быть поле типа «родительская подсистема» (в документации EDT — ParentSubsystem). Формат в XML может быть объект с `item` или аналог (как у других ссылок в `extractPropertiesFromMdo`). Нужно в парсере явно извлекать это свойство и сохранять в `properties.parentSubsystemRef` (нормализованно: строка имени подсистемы или id, достаточная для сопоставления с узлами подсистем при построении дерева).
- Контракт парсера: после разбора типа Subsystems парсер возвращает плоский список узлов подсистем с заполненным `parentSubsystemRef` (если есть родитель); затем отдельный шаг (в парсере или в провайдере дерева) строит из этого списка иерархию и подвешивает под узел «Subsystems» только корневые.

**Конфигуратор (cf/xml):**

- В XML подсистемы иерархия задаётся **списком дочерних подсистем** в `ChildObjects`:
  ```xml
  <ChildObjects>
    <Subsystem>НСИЗакупок</Subsystem>
    <Subsystem>РасчетыСПоставщиками</Subsystem>
    ...
  </ChildObjects>
  ```
  Т.е. в файле подсистемы перечислены **дочерние** подсистемы (не родитель). Алгоритм построения дерева для конфигуратора:
  1. **Обнаружить все** XML подсистем (в т.ч. вложенные по путям вида `Subsystems/Закупки/Subsystems/Закупки.xml`), прочитать каждый, извлечь имя (Name), uuid (если есть), ChildObjects → `childSubsystemNames`, Content; сохранить в узле `filePath` = путь к этому XML.
  2. Корневые подсистемы = те, чьё имя **ни разу** не встречается в списке дочерних ни у одной другой подсистемы (при неоднозначности — по пути к файлу: корневая та, чей XML лежит прямо в `Subsystems/Имя.xml` или в корне типа).
  3. При сборке дерева **присваивать уникальный id** (path-based: корневая `Subsystems.${name}`, дочерняя `${parent.id}.${name}`; при повторяющихся именах у разных родителей id автоматически различаются).
  4. Для каждой подсистемы в `children` класть узлы подсистем, соответствующие её `childSubsystemNames`; сопоставление «имя → узел» при нескольких узлах с одним именем — по контексту (родитель уже известен по дереву, дочерний узел — тот, чей filePath лежит под каталогом родителя или однозначно выводится из структуры каталогов выгрузки).
- Общий алгоритм из п.2 (по родителю) применяется в EDT; для конфигуратора используется эквивалент по списку детей (инвертированная связь). Из-за повторяющихся имён при конфигураторе важно опираться на **путь к файлу** и присвоение **id по пути в дереве**, чтобы в дереве и в фильтре однозначно определялась «та самая» подсистема.

### 4. Фильтр по подсистеме

- Логика **не меняется**: «принадлежность узла выбранной подсистеме» = узел сам является этой подсистемой, или является предком этой подсистемы в дереве (путь от узла вверх по `parent` до выбранной подсистемы), или входит в Content выбранной подсистемы (и при необходимости в Content всех её потомков — если решим учитывать вложенные подсистемы).
- С иерархией подсистем цепочка `parent` для узла подсистемы станет правильной (Configuration → Subsystems → Корневая → … → Выбранная), поэтому проверка «узел — предок выбранной подсистемы» будет работать корректно.
- Рекомендация: при фильтре по подсистеме считать видимыми объекты из Content **этой** подсистемы и из Content **всех её дочерних подсистем** (рекурсивно), чтобы «Фильтр по подсистеме X» показывал всё, что логически входит в X. Это уточнение поведения, а не изменение контракта фильтра.

### 5. Отображение и ленивая загрузка

- При lazy loading: при первом запросе `getChildren` для узла типа «Subsystems» парсер должен вернуть уже **иерархический** список (только корневые подсистемы). Это может потребовать при первой загрузке типа Subsystems загружать все подсистемы (чтобы прочитать родителя), строить дерево и отдавать только корневые.
- При раскрытии узла подсистемы дочерние узлы — в первую очередь дочерние подсистемы (и при необходимости другие дочерние элементы по формату). Загрузка дочерних подсистем — по уже построенному дереву (без повторного чтения MDO), если дерево подсистем строится один раз при загрузке типа.

## Контракты (кратко)

| Участник | Обязательство |
|----------|----------------|
| **Парсер (EDT)** | Для каждого узла подсистемы заполнять `properties.parentSubsystemRef` из Subsystem.mdo (ParentSubsystem или аналог); при сборке типа Subsystems возвращать либо плоский список с этими полями, либо сразу дерево (только корневые в `children` узла типа). |
| **Парсер (Конфигуратор)** | Обходить все XML подсистем (включая вложенные пути). Из каждого извлекать ChildObjects → `childSubsystemNames`. Присваивать каждому узлу **уникальный id** (path-based: `Subsystems.${name}` для корневых, `${parent.id}.${name}` для дочерних). Строить дерево; при нескольких узлах с одним именем сопоставление родитель–ребёнок по путям к файлам. |
| **TreeDataProvider** | Считать, что у узла типа Subsystems в `children` только корневые подсистемы; у каждой подсистемы в `children` — дочерние подсистемы; у каждого узла подсистемы **уникальный id**. При фильтре «какая подсистема» определяется выбранным узлом (его id и filePath), а не строкой имени. Использовать существующую логику (предок + Content по выбранному узлу); при необходимости — Content всех потомков в дереве подсистем. |
| **TreeNode** | Без изменения структуры; опциональное свойство `properties.parentSubsystemRef` только для подсистем. |

## Последствия

- **Плюсы:** дерево подсистем совпадает с 1С; фильтр по подсистеме и отображение вложенности работают предсказуемо; одна модель для EDT и Конфигуратора.
- **Риски:** при первой загрузке типа Subsystems может понадобиться прочитать все Subsystem.mdo (для родителя), а не только имена папок — приемлемо при ленивой загрузке типа.
- **Тесты:** обновить/добавить тесты на дерево подсистем (корневые/дочерние), на фильтр по вложенной подсистеме и на разрешение родителя по имени/id.

## Открытые моменты (для реализации)

1. Точный формат поля родителя в Subsystem.mdo (имя тега и структура в XML) — уточнить по реальному MDO или документации EDT.
2. ~~Конфигуратор: где в выгрузке хранится родитель подсистемы~~ — **решено:** в XML конфигуратора в каждом файле подсистемы в `ChildObjects` перечислены дочерние подсистемы (`<Subsystem>Имя</Subsystem>`). Нужна функция извлечения (например `extractChildSubsystems` в xmlChildObjects) и построение дерева по спискам детей.
3. **Повторяющиеся имена подсистем:** в дереве и в фильтре «какая из них» определяется только уникальным **id** узла (и его filePath для загрузки Content). Реализация должна присваивать path-based id при построении дерева и при сопоставлении «родитель → дочерний узел по имени» учитывать контекст (путь к файлу вложенной подсистемы).
4. Content дочерних подсистем: явно зафиксировать в коде/спеках, что при фильтре по подсистеме показываются объекты из Content этой подсистемы и всех её потомков в дереве подсистем.

После ревью дизайна — переход к фазе Planning и Implementation (парсеры + провайдер дерева).
