# Infobase Manager — Дизайн фичи

> **Статус:** Draft  
> **Автор:** AI Design Agent  
> **Дата:** 2026-03-27

## 1. Цель

Дать пользователю расширения CDT 41 возможность управлять информационными базами 1С прямо из VS Code:
- Создавать / добавлять / удалять базы из списка
- Загружать конфигурацию в базу (import)
- Выгружать конфигурацию из базы (export / dump)
- Запускать 1С:Предприятие и Конфигуратор
- Импортировать списки баз из `.v8i` файлов

Всё это — через утилиту автономного сервера **ibcmd** (8.3.27+).

---

## 2. Контекст

### 2.1 Текущее состояние

- Расширение уже использует `ibcmd` для `config check` и `config import` (gate подсистем, матрица e2e).
- Путь к `ibcmd` и конфиг ИБ задаются через **env** (`IBCMD_PATH`, `IBCMD_INFOBASE_CONFIG`).
- Нет UI для управления несколькими базами — только одна «текущая» ИБ через YAML.
- Нет импорта `.v8i`.

### 2.2 Целевая аудитория

| Персона | Потребность |
|---------|-------------|
| **Разработчик 1С** | Быстро переключаться между базами, загружать изменения, открывать конфигуратор |
| **Техлид / Архитектор** | Поддерживать несколько тестовых баз, сравнивать версии |
| **DevOps / CI** | Автоматизация через команды VS Code (tasks, CLI) |

## 3. Use-Cases

### UC-01: Первоначальная настройка

**Актор:** Разработчик  
**Предусловие:** Расширение установлено, ibcmd доступен в системе  
**Основной сценарий:**

1. Пользователь открывает панель «Infobase Manager» (Activity Bar или команда)
2. Видит пустой список баз и подсказку «Укажите путь к ibcmd в настройках»
3. Переходит в Settings → `1cMetadataTree.ibcmd.path`
4. Указывает путь (или расширение находит автоматически в PATH / стандартных местах)
5. Панель обновляется, показывает «Нет баз. Добавьте базу или импортируйте список.»

**Альтернативы:**
- 3a. ibcmd найден автоматически → шаг 4 пропускается

---

### UC-02: Создание новой базы

**Актор:** Разработчик  
**Предусловие:** ibcmd настроен  
**Основной сценарий:**

1. Пользователь нажимает «+ Создать базу» в панели
2. Открывается Quick Pick: «Файловая» / «Серверная» / «Веб»
3. Выбирает «Файловая»
4. Диалог выбора папки для новой базы
5. Ввод имени базы (Quick Input)
6. Расширение вызывает `ibcmd infobase create --db-path=...`
7. База появляется в списке
8. Показывается уведомление: «База {name} создана»

**Постусловие:** База создана на диске, добавлена в список расширения, пользователь уведомлён

**Альтернативы:**
- 3a. Выбирает «Серверная» → ввод параметров сервера, создание через ibcmd
- 3b. Выбирает «Веб» → ввод URL веб-клиента, база добавляется в список (без создания через ibcmd)

---

### UC-03: Добавление существующей базы

**Актор:** Разработчик  
**Основной сценарий:**

1. Нажимает «+ Добавить существующую»
2. Quick Pick: «Файловая» / «Серверная» / «Веб» / «Из строки подключения»
3. Для файловой: диалог выбора папки с существующей базой
4. Для серверной: ввод сервера, имени базы, опционально логин/пароль
5. Для веб: ввод URL веб-клиента, имя базы
6. Расширение проверяет доступность (для file/server — `ibcmd infobase info`)
7. База добавляется в список

---

### UC-04: Импорт списка баз из .v8i

**Актор:** Разработчик  
**Предусловие:** Есть файл `.v8i` (экспорт из 1С:Предприятие или с сервера)  
**Основной сценарий:**

1. Нажимает «Импорт из .v8i»
2. Диалог выбора файла (фильтр `*.v8i`)
3. Расширение парсит файл, показывает превью списка баз
4. Пользователь выбирает какие базы импортировать (checkbox list)
5. Выбранные базы добавляются в список

**Альтернативы:**
- 3a. Файл повреждён / неизвестная кодировка → сообщение об ошибке

---

### UC-05: Загрузка конфигурации в базу (Import)

**Актор:** Разработчик  
**Предусловие:** В workspace открыта выгрузка конфигурации, база в списке  
**Основной сценарий:**

1. Правый клик на базе → «Загрузить конфигурацию...»
2. Quick Pick: выбор источника
   - «Из текущего workspace» (если есть `Configuration.xml`)
   - «Выбрать папку...»
3. Подтверждение: «Загрузить конфигурацию в базу {name}? Это перезапишет текущую.»
4. Расширение вызывает `ibcmd infobase config import --config=... <path>`
5. Progress notification с выводом ibcmd
6. По завершении: «Конфигурация загружена» или ошибка

**Альтернативы:**
- 4a. База заблокирована → сообщение, предложение закрыть сеансы

---

### UC-06: Выгрузка конфигурации из базы (Export)

**Актор:** Разработчик  
**Основной сценарий:**

1. Правый клик на базе → «Выгрузить конфигурацию...»
2. Диалог выбора целевой папки
3. Опции: «Перезаписать если существует» (checkbox)
4. Расширение вызывает `ibcmd infobase config export --config=... --out=<path>`
5. Progress notification
6. По завершении: предложение открыть папку в VS Code

---

### UC-07: Запуск 1С:Предприятие

**Актор:** Разработчик  
**Основной сценарий:**

1. Правый клик на базе → «Открыть в 1С:Предприятие»
2. Расширение формирует строку подключения и запускает `1cv8.exe` (или `1cv8c`)
3. Окно 1С открывается

**Настройки:** Путь к `1cv8.exe` в settings

---

### UC-08: Запуск Конфигуратора

**Актор:** Разработчик  
**Основной сценарий:**

1. Правый клик на базе → «Открыть Конфигуратор»
2. Расширение запускает `1cv8.exe /F"..." DESIGNER` (файловая) или аналог для серверной
3. Конфигуратор открывается

---

### UC-09: Проверка конфигурации (Config Check)

**Актор:** Разработчик  
**Основной сценарий:**

1. Правый клик на базе → «Проверить конфигурацию»
2. Расширение вызывает `ibcmd infobase config check --config=...`
3. Результат в Output Channel или открывается отчёт

---

### UC-10: Удаление базы из списка

**Актор:** Разработчик  
**Основной сценарий:**

1. Правый клик на базе → «Удалить из списка»
2. Подтверждение: «Удалить базу {name} из списка? Файлы базы не будут удалены.»
3. База удаляется из списка расширения

**Альтернатива:**
- 2a. Опция «Также удалить файлы базы» (только для файловых, с дополнительным подтверждением)

## 4. Модель данных

### 4.1 InfobaseEntry

```typescript
interface InfobaseEntry {
  id: string;                    // UUID
  name: string;                  // Отображаемое имя
  type: 'file' | 'server' | 'web'; // Тип базы
  
  // Для файловой базы
  filePath?: string;             // Абсолютный путь к папке базы
  
  // Для серверной базы
  server?: string;               // Имя сервера 1С
  database?: string;             // Имя базы на сервере
  
  // Для web-базы
  webUrl?: string;               // URL веб-клиента
  
  // Аутентификация (опционально)
  user?: string;
  // password хранится в SecretStorage VS Code
  
  // Настройки запуска (опциональные, см. секцию 15)
  launchSettings?: {
    platformVersion?: string;    // "8.3.24.1467"
    clientType?: 'thin' | 'thick' | 'web';
    bitness?: '32' | '64';
  };
  
  // Метаданные
  createdAt: string;             // ISO date
  lastUsedAt?: string;           // ISO date
}
```

> **Примечание:** Полная логика настроек запуска описана в секции 15.

### 4.1.1 Замечания к реализации Phase 1 (согласование с кодом)

Чтобы дальнейшие фазы (UI, команды, привязки) не расходились с фактической моделью в репозитории, зафиксированы **дополнения** к интерфейсу §4.1:

| Поле / решение | Зачем |
|----------------|--------|
| **`hasStoredPassword: boolean`** | В JSON записи пароля нет (§4.2); без флага нельзя однозначно знать, вызывать ли `SecretStorage`, не перебирая ключи. Семантика совместима с дизайном («пароль только в SecretStorage»). |
| **`ibcmdConfigYamlPath?: string`** (опционально) | Для `ibcmd` нужен аргумент `--config` к **YAML** конфигурации ИБ; он не всегда выводим из одного `filePath` (разные раскладки каталогов, наследие хранилища «только путь к yaml»). Для типичной файловой базы достаточно `filePath`; явный YAML — когда путь к каталогу и путь к yaml расходятся. |
| **Устаревшие ключи `1cInfobaseManager.*`** | Краткий промежуток внедрения использовал другой префикс; в рантайме читается сначала **`1cMetadataTree.*`** (§5), при пустом значении — fallback на legacy, чтобы не ломать уже сохранённые настройки. Новые сценарии ориентируются только на §5. |
| **Команда настройки ibcmd** | В коде может использоваться id вида `…infobaseManager.showIbcmdSetup` до появления полного набора команд из §7 (`…infobases.*`). При реализации дерева баз — либо алиас, либо переименование с сохранением обратной совместимости в `package.json`. |
| **Модули ibcmd** | Дизайн §8.1 допускает один файл `ibcmdService.ts`; фактически допустимо несколько файлов под `services/ibcmd/` с тонким barrel — внешний контракт для остального расширения тот же. |

Если в Phase 2+ появится жёсткое правило «всегда выводить yaml из `filePath`», поле `ibcmdConfigYamlPath` можно убрать после миграции данных и обновления сценариев добавления базы.

### 4.2 Хранение

- **Список баз:** `globalState` (глобальный пул на все проекты). В реализации Phase 1 используется версионируемый ключ (например `1cMetadataTree.infobases.v1`) и миграция с прежнего ключа `1cInfobaseManager.v1` при первом обращении.
- **Привязки:** `.vscode/infobase-bindings.json` (per-workspace)
- **Пароли:** VS Code `SecretStorage` API (ключи стабильно привязаны к `id` записи; при миграции хранилища — перенос секретов на новый префикс ключей).

## 9. Формат .v8i

```ini

Connect=File="C:\Bases\Demo_UT";
```

- `Connect=` — строка подключения (парсить отдельно)
- Остальные поля — метаданные (опционально)

## 10. Риски и митигации

| Риск | Вероятность | Влияние | Митигация |
|------|-------------|---------|-----------|
| ibcmd недоступен | Средняя | Высокое | Graceful degradation, подсказки по установке |
| Разные версии ibcmd | Средняя | Среднее | Проверка версии, документация совместимости |
| Блокировки баз | Высокая | Среднее | Информативные сообщения, продолжение на следующую базу |
| Большие базы — долгий import | Высокая | Среднее | Progress с отменой, таймауты в настройках |
| Пароли в памяти | Низкая | Высокое | SecretStorage, не логировать |
| Ошибка на одной базе при массовой раскатке | Высокая | Среднее | Логировать ошибку, продолжать остальные, итоговый отчёт |
| Путь к 1cv8.exe не найден | Высокая | Среднее | Автоопределение, подсказка настроить в settings |
| Удаление базы с привязками | Средняя | Среднее | Каскадное удаление из привязок, уведомление |
| Дубликат базы при добавлении | Средняя | Низкое | Валидация при создании, сообщение «уже существует» |
| Режим copy — большой проект, долгое копирование | Средняя | Среднее | Инкрементальное копирование, исключение node_modules/dist |
| Режим block — пользователь забыл сохранить | Средняя | Среднее | Предупреждение перед блокировкой, auto-save |

## 12. Привязка баз к конфигурации (Binding)

### 12.1 Концепция

Корневой элемент **Configuration** в дереве CDT 41 получает возможность привязать одну или несколько информационных баз. Это позволяет:
- Быстро раскатывать изменения конфигурации в привязанные базы
- Поддерживать как единичную, так и массовую раскатку

### 12.2 UC-11: Привязка баз к конфигурации

**Актор:** Разработчик  
**Предусловие:** Открыт workspace с выгрузкой конфигурации, есть базы в Infobase Manager  
**Основной сценарий:**

1. Правый клик на **Configuration** в дереве CDT 41 → «Привязать базы...»
2. Открывается диалог «Привязка баз»
3. Слева — список привязанных баз (изначально пуст)
4. Справа — кнопки:
   - «Добавить из списка» → Quick Pick с существующими базами
   - «Создать новую...» → переход к мастеру создания базы (UC-02)
   - «Добавить существующую...» → переход к мастеру добавления (UC-03)
5. Пользователь добавляет базы в список привязки
6. Стрелками ↑↓ управляет порядком баз
7. Нажимает «Сохранить»

**Постусловие:** Привязка сохранена, Configuration в дереве показывает индикатор привязки

### 12.3 Режим раскатки: единичная vs массовая

#### Флаг «Массовая раскатка»

- **Включён (по умолчанию при >1 базы):** все базы в списке активны, раскатка идёт последовательно на каждую
- **Выключен:** активна только **первая** база в списке, остальные затенены

#### Поведение при переключении флага

| Действие | Результат |
|----------|-----------|
| Снять флаг | Список затеняется, первая база подсвечена. Показывается предупреждение: «Раскатка будет выполнена только в базу: {имя первой базы}» |
| Поднять флаг | Список становится активным. Базы из списка **не удаляются** — сохраняется весь набор |

#### Персистентность

- Флаг массовой раскатки хранится **per-configuration** в `workspaceState` (или `.vscode/infobase-bindings.json`)
- При открытии workspace флаг восстанавливается

### 12.5 UC-12: Раскатка конфигурации в привязанные базы

**Актор:** Разработчик  
**Предусловие:** Конфигурация имеет привязанные базы  
**Основной сценарий:**

1. Правый клик на **Configuration** → «Раскатать в базу» (или «Раскатать в базы» при массовом режиме)
2. Подтверждение с перечнем целевых баз
3. Раскатка выполняется **последовательно** в порядке списка привязки
4. Progress notification показывает: текущую базу, номер из общего числа, вывод ibcmd
5. По завершении каждой базы — лог результата (успех/ошибка)
6. По завершении всех — итоговое уведомление: «Раскатка завершена: N успешно, M с ошибками»

**Альтернативы:**
- 3a. База заблокирована / ошибка ibcmd → логируется ошибка, **продолжаем** на следующую базу
- 4a. Пользователь отменяет → текущая операция завершается, остальные пропускаются

**Постусловие:** Конфигурация загружена в доступные базы, пользователь видит итог

**Примечание:** Иконка и текст команды меняются в зависимости от флага массовой раскатки:
- Массовая вкл: «Раскатать в базы» (иконка множественная)
- Массовая выкл: «Раскатать в базу» (иконка единичная)

### 12.7 Индикация в дереве

### 12.8 Идентификация конфигурации (configPath)

**Проблема:** Абсолютный путь ломается при переносе workspace.

**Решение:** Использовать **относительный путь от workspace root** + **workspace folder name** для multi-root:

```typescript
interface ConfigurationBinding {
  /** Имя workspace folder (для multi-root) */
  workspaceFolder: string;
  
  /** Относительный путь к Configuration.xml от корня workspace folder */
  configRelativePath: string;
  
  /** Список ID привязанных баз (порядок важен) */
  infobaseIds: string[];
  
  /** Флаг массовой раскатки */
  massDeployment: boolean;
}
```

При открытии workspace:
1. Найти workspace folder по имени
2. Резолвить полный путь: `workspaceFolderPath + configRelativePath`
3. Если не найдено — показать предупреждение, предложить перепривязать

## 14. Инварианты и контракты

### 14.1 Уникальность базы

- При создании/добавлении базы — валидация на дубликат
- Критерий дубликата:
  - Для `file`: совпадение `filePath` (нормализованный)
  - Для `server`: совпадение `server` + `database`
  - Для `web`: совпадение `webUrl`
- При обнаружении дубликата: «База уже существует в списке: {name}»

### 14.2 Удаление базы

- При удалении из **общего списка** → **каскадное удаление** из всех привязок
- При удалении из **привязки** → база остаётся в общем списке

### 14.3 Порядок в привязке

- Порядок `infobaseIds[]` определяет:
  - Порядок раскатки (сверху вниз)
  - «Первую» базу при выключенном флаге массовой раскатки

### 14.4 Персистентность флага

- Флаг `massDeployment` сохраняется **только по кнопке «Сохранить»**
- Закрытие диалога без сохранения — изменения отменяются

## 15. Настройки запуска базы

### 15.1 Расширенная модель InfobaseEntry

```typescript
interface InfobaseEntry {
  id: string;
  name: string;
  type: 'file' | 'server' | 'web';
  
  // Подключение (по типу)
  filePath?: string;
  server?: string;
  database?: string;
  webUrl?: string;
  
  // Аутентификация
  user?: string;
  // password в SecretStorage
  
  // Настройки запуска (опциональные)
  launchSettings?: {
    /** Версия платформы (например "8.3.24.1467") */
    platformVersion?: string;
    
    /** Режим запуска */
    clientType?: 'thin' | 'thick' | 'web';
    
    /** Разрядность */
    bitness?: '32' | '64';
  };
  
  // Метаданные
  createdAt: string;
  lastUsedAt?: string;
}
```

### 15.2 Логика определения параметров запуска

| Параметр | Если задан | Если не задан |
|----------|------------|---------------|
| `platformVersion` | Запускать указанную версию | Последняя установленная |
| `clientType` | Запускать указанный тип | Тонкий клиент |
| `bitness` | Запускать указанную разрядность | Системная (или найденная) |

### 15.3 Поиск платформы

1. Если `platformVersion` задана — искать точное совпадение
2. Если не найдена — ошибка с предложением установить или выбрать другую
3. Если не задана — брать последнюю из установленных
4. Пути поиска: стандартные каталоги установки 1С + PATH

## 16. Открытые вопросы (решения)

### 16.1 Хранение списка баз

**Вопрос:** workspace-level vs global?

**Анализ вариантов:**

| Вариант | Плюсы | Минусы |
|---------|-------|--------|
| **A: Только workspace** (`.vscode/infobases.json`) | Версионируется с проектом, изоляция между проектами | Дублирование баз между проектами, не шарится |
| **B: Только global** (`globalState`) | Одна точка правды, базы доступны везде | Нет изоляции, конфликты между проектами, не версионируется |
| **C: Оба уровня** | Гибкость: общие базы + проектные | Сложность UI, путаница «откуда база» |
| **D: Global + workspace override** | Общий пул + проектные настройки привязок | Баланс гибкости и простоты |

**Решение: Вариант D** ✅

- Список баз хранится **глобально** (`globalState`) — один пул на все проекты
- Привязки к конфигурациям хранятся **per-workspace** (в `.vscode/infobase-bindings.json`)
- Причина: базы — это физические сущности (папки, серверы), они не зависят от проекта. Привязки — это связь конкретного проекта с базами.

### 16.3 Серверные базы — права и HTTPS

**Доресерч:**

#### Права для операций с конфигурацией

| Операция | Требуемые права |
|----------|-----------------|
| `ibcmd infobase config import` | Права СУБД (запись в базу), НЕ требует прав кластера |
| `ibcmd infobase config export` | Права СУБД (чтение), НЕ требует прав кластера |
| `ibcmd infobase create` (серверная) | Права администратора кластера + права СУБД |
| `ibcmd infobase info` | Права СУБД (чтение) |

**Вывод:** Для основных операций (import/export) достаточно прав СУБД. Права кластера нужны только для создания новой серверной базы.

#### HTTPS

- `ibcmd` поддерживает подключение к серверу 1С через стандартный порт (1541)
- HTTPS используется для веб-сервисов 1С, не для ibcmd
- Для серверных баз указывается: `Srvr="server:port";Ref="database"`

**Решение:** 
- Документировать требования к правам СУБД
- HTTPS не релевантен для ibcmd (только для web-баз)
- При создании серверной базы предупреждать о необходимости прав администратора кластера

### 16.4 Расширения 1С

**Решение:** Поддержка расширений планируется. В дереве CDT 41 расширения живут отдельным root-элементом. Каждое расширение может иметь свои привязки к базам.

На текущем этапе:
- Расширения обрабатываются по общим правилам (как конфигурация)
- Полная поддержка (отдельные биндинги, загрузка расширений без основной конфигурации) — в Phase 4+

### 16.5 Блокировка разработки при раскатке

**Решение:** Реализовать оба варианта, выбор через настройку.

```jsonc
{
  // Режим раскатки
  "1cMetadataTree.deploy.mode": "copy" | "block"
}
```

| Режим | Поведение |
|-------|-----------|
| `copy` | Создаётся временная копия **папки конфигурации** (от `Configuration.xml`), раскатка из копии. Разработка не блокируется. |
| `block` | Блокируются все редакторы/editors на время раскатки. Показывается progress overlay. |

**По умолчанию:** `copy` (менее инвазивный)

**Реализация `copy`:**
- Копируется только дерево выгрузки конфигурации (папка с `Configuration.xml` и вложенные)
- Копия создаётся в temp-директории
- После раскатки копия удаляется

**Реализация `block`:**
- Использовать `vscode.window.withProgress` с `cancellable: true`
- Установить `editor.readOnly` на время операции (или использовать custom overlay)
- Предупреждение перед началом: «Редактирование будет заблокировано на время раскатки»
