## Дизайн записи прав в редакторе ролей

### 1. Целевые файлы для записи

1. **Случай A: права хранятся в отдельном Rights.xml**  
   - Открытый файл в VS Code: как правило `Roles/ИмяРоли.xml` (EDT‑описание роли или выгрузка из конфигуратора).  
   - Права при этом лежат в `Roles/ИмяРоли/Ext/Rights.xml` с корневым элементом `<Rights xmlns="http://v8.1c.ru/8.2/roles">`.  
   - **Фактическая цель записи прав**: всегда `Roles/ИмяРоли/Ext/Rights.xml`.  
   - **Получение пути к Rights.xml по `RoleModel.filePath`**:  
     - `roleFilePath = currentRoleModel.filePath` (ожидается путь к `Roles/ИмяРоли.xml`).  
     - `roleDir = dirname(roleFilePath)` → `Roles`.  
     - `roleName = basename(roleFilePath, extname(roleFilePath))` → `ИмяРоли`.  
     - `rightsPath = join(roleDir, roleName, "Ext", "Rights.xml")`.  
   - **Запись в `Roles/ИмяРоли.xml` не выполняется**: структура файла описания роли не меняется; изменяется только `Ext/Rights.xml`.

2. **Случай B: права хранятся внутри Role.xml**  
   - Открытый файл в VS Code: `Roles/ИмяРоли/Role.xml` (или другой путь, где сам `Role.xml` содержит узел `Role/Rights/...`).  
   - `currentRoleModel.filePath` указывает на этот файл, и `RoleXmlParser.extractRights` получает права именно оттуда.  
   - **Фактическая цель записи**: **тот же файл**, что и открыт пользователем, то есть `currentRoleModel.filePath`.  
   - Если дополнительно существует `Roles/ИмяРоли/Ext/Rights.xml`, текущий редактор прав его **не модифицирует**.

### 2. Запись для случая B (Role.xml)

1. **Текущий поток сохранения** (оставляем, только аккуратно завершаем):  
   - `rolesRightsEditorProvider.handleSave`:  
     - валидирует `RoleModel` через `RightsValidator`;  
     - сериализует модель: `RoleXmlSerializer.serializeToXml(roleModel)` → XML-строка в формате Designer (`Role/Rights/...`);  
     - записывает результат в `currentRoleModel.filePath`.  

2. **Требования к записи**  
   - **Кодировка**: UTF-8 без BOM (совместимо с текущим чтением/записью).  
   - **Атомарность**:  
     - Сериализовать XML в строку (полный документ).  
     - Записать строку во **временный файл** в той же директории, что и целевой `Role.xml`, например:  
       - `tempPath = targetPath + ".tmp"` или с уникальным суффиксом.  
     - После успешной записи и закрытия временного файла выполнить `rename`/`move` поверх целевого `Role.xml`.  
     - При ошибке записи/переименования удалить временный файл (если существует) и **не изменять** исходный `Role.xml`.  
   - **Обработка ошибок**:  
     - Любая ошибка сериализации/записи → показать пользователю `vscode.window.showErrorMessage` с понятным текстом (формат файла, путь, базовое описание ошибки).  
     - Сообщить VS Code о неуспешном сохранении (send `saveError` / отклонить `CustomTextEditor` save).  
   - **Edge cases**:  
     - **Read-only файл / отсутствие прав записи**:  
       - Ошибка при записи/переименовании интерпретируется как отказ ОС; показать сообщение вида «Не удалось сохранить Role.xml: файл только для чтения или нет прав на запись».  
       - Данные в памяти не теряются; UI остаётся открытым.  
     - **Внешние изменения файла**:  
       - Базовое поведение: продолжаем писать поверх файла (VS Code уже мог предупредить пользователя об изменении).  
       - В первой итерации **не реализуем** сложную детекцию конфликтов; полагаемся на стандартный механизм VS Code и VCS.  

3. **Бэкапы для Designer**  
   - Обязательное создание `.bak` не внедряем на первом шаге (получится дублирование с VCS и может захламлять репозиторий).  
   - При необходимости можно будет добавить опциональное создание `Role.xml.bak` **до** атомарной перезаписи, но эта функциональность не является частью текущего дизайна.

### 3. Запись для EDT (Ext/Rights.xml) — общий подход

1. **Почему нельзя использовать `RoleXmlSerializer` напрямую**  
   - `RoleXmlSerializer` формирует Designer-структуру (`Role/Rights/Catalog/Object/Name/Read/…`).  
   - EDT использует **другой формат**: `Rights` → `object` → `name`/`right` (`name`/`value` + `restrictionByCondition` и блоки `restrictionTemplate`).  
   - Простая перезапись `Rights.xml` данным форматом приведёт к полной потере:  
     - всех `restrictionByCondition` (условий доступа);  
     - всех `restrictionTemplate`;  
     - любых дополнительных/неизвестных прав, не управляемых текущим редактором.  

2. **Целевой алгоритм (A–C)**  
   - **A. Парсинг существующего EDT Rights.xml**  
     - Определить путь к `rightsPath` (см. раздел 1.1).  
     - Если файл существует:  
       - Считать XML, распарсить в структуру, которая **сохраняет весь DOM/дерево** (узлы, атрибуты, порядок по возможности):  
         - корневой элемент `Rights` с атрибутами/подэлементами;  
         - все `object` и их содержимое;  
         - все `restrictionTemplate` и другие узлы.  
       - При ошибке парсинга (некорректный XML):  
         - показать пользователю сообщение об ошибке с указанием пути;  
         - сохранить файла **не выполнять**;  
         - вернуть ошибку сохранения в VS Code.  
     - Если файл **не существует**:  
       - Создать **новую** модель документа с минимально необходимой структурой:  
         - корневой `<Rights>` с корректным `xmlns` и базовыми подэлементами:  
           - `setForNewObjects` (по умолчанию `false`);  
           - `setForAttributesByDefault` (по умолчанию `true`);  
           - `independentRightsOfChildObjects` (по умолчанию `false`);  
         - без `restrictionTemplate` (они появятся только если пользователь создаст/скачает шаблон другими средствами).  

   - **B. Слияние `RoleModel.rights` с EDT-структурой**  
     - На входе:  
       - `roleModel.rights: RightsMap` — словарь `objectFullName` → `ObjectRights` (то, что редактирует UI).  
       - `rightsDom` — существующая/новая DOM-модель `Rights.xml`.  
     - Подход: применять изменения **только к тем узлам**, которыми управляет редактор, и не трогать остальную структуру.  

3. **Правила поиска/создания объектов (`<object>`)**

1. **Поиск объекта по имени**  
   - Для каждого `objectFullName` из `RoleModel.rights`:  
     - Ищем в `rightsDom` элемент `<object>`, внутри которого есть дочерний `<name>` с текстом **точно равным** этому `objectFullName`.  
     - Сравнение выполняется как строковое (без нормализации регистров/пространств имён, учитываем текущий формат EDT).  

2. **Обновление существующего `<object>`**  
   - Если такой `<object>` найден:  
     - Используем его как целевой контейнер для прав;  
     - Не меняем порядок `<object>` в документе и не удаляем существующие узлы внутри (кроме точечного изменения `<value>` в известных правах или добавления новых `<right>`).  

3. **Создание нового `<object>`**  
   - Если `<object>` для `objectFullName` **не найден**:  
     - Создаём новый элемент `<object>` в конце списка существующих `object` (перед первыми `restrictionTemplate`, если они есть).  
     - Внутри создаём:  
       - `<name>` с текстом `objectFullName`;  
       - набор `<right>` согласно `ObjectRights` из модели (см. правила ниже).  
     - Остальные свойства `Rights` (флаги, шаблоны) **не меняются**.

4. **Правила обновления/добавления `<right>` внутри объекта**

1. **Область ответственности редактора**  
   - Редактор оперирует **набором известных прав**, соответствующих колонкам UI (например: `Read`, `Insert`, `Update`, `Delete`, `View`, специфические флаги и т.п.).  
   - Все прочие права в `Rights.xml`, которых **нет** в `ObjectRights` для текущего объекта, считаются **вне зоны ответственности текущего редактора** и должны оставаться без изменений.  

2. **Поиск существующих прав**  
   - Для каждого права `rightName` из `ObjectRights` выбранного объекта:  
     - Внутри `<object>` ищем первый `<right>`, содержащий дочерний `<name>` с текстом `rightName`.  
     - Если найдено несколько `<right>` с одинаковым `name`, изменяем только **первый** (предполагаем, что дубликаты — ошибка, но не решаем её здесь).  

3. **Обновление значения известного права**  
   - Если `<right>` с нужным `name` найден:  
     - Находим дочерний элемент `<value>` и обновляем его текстовое содержимое на `"true"` или `"false"` в соответствии со значением в `ObjectRights`.  
     - Все остальные дочерние элементы `<right>` (в том числе `<restrictionByCondition>` с вложенным `<condition>` и любые дополнительные теги) **сохраняются без изменений**.  
     - Структура и порядок дочерних элементов `<right>` не трогаются.  

4. **Добавление нового `<right>`**  
   - Если `<right>` с указанным `rightName` **не найден**:  
     - Создаём новый элемент `<right>` и добавляем его в конец списка `right` внутри текущего `<object>`.  
     - Внутри создаём:  
       - `<name>` с текстом `rightName`;  
       - `<value>` с `"true"`/`"false"` по данным из `ObjectRights`.  
     - **Не создаём** `<restrictionByCondition>` автоматически; если пользователь хочет ограничение по условию, оно должно быть добавлено другими инструментами (не этим редактором).  

5. **Сохранение неизвестных прав**  
   - `<right>`-элементы, для которых `name` отсутствует в `ObjectRights` текущего `objectFullName`, полностью **сохраняются**:  
     - не удаляются;  
     - не меняется `value`;  
     - не затрагиваются `restrictionByCondition` и любые другие вложенные узлы.  

5. **Глобальные элементы `Rights`**  
   - Элементы верхнего уровня:  
     - `setForNewObjects`;  
     - `setForAttributesByDefault`;  
     - `independentRightsOfChildObjects`;  
     - все `restrictionTemplate`;  
     - любые другие глобальные узлы, которые может добавлять EDT.  
   - **Никогда не изменяются** редактором прав:  
     - значения не перезаписываются;  
     - новые элементы не удаляются;  
     - порядок следования не меняется (кроме возможного добавления новых `<object>` перед первым `restrictionTemplate`).  

### 4. Сериализация EDT Rights.xml назад в файл

1. **Формирование XML**  
   - После применения всех изменений к `rightsDom` выполняется сериализация обратно в XML-строку.  
   - Требования:  
     - Сохранить XML-декларацию `<?xml version="1.0" encoding="UTF-8"?>`.  
     - Сохранить пространство имён и атрибуты корневого `Rights` (`xmlns`, `xmlns:xs`, `xmlns:xsi`, `xsi:type`, `version` и т.п.) без изменений.  
     - Форматирование (отступы/переносы строк) можно не воспроизводить полностью, но желательно использовать единый стиль (например, с отступами в табуляции/пробелах, если библиотека это поддерживает).  

2. **Атомарная запись**  
   - Аналогично Designer:  
     - Пишем в `rightsPath + ".tmp"` или аналогичный временный путь в той же директории;  
     - После успешной записи выполняем `rename` на целевой `rightsPath`;  
     - При ошибке удаляем временный файл и не трогаем исходный `Rights.xml`.  
   - Кодировка: UTF-8 без BOM.  

3. **Ошибки записи/сериализации**  
   - Если сериализация `rightsDom` в строку завершилась с ошибкой — показываем сообщение пользователю и **не пытаемся** записать файл.  
   - Если ошибка произошла при записи/переименовании — сообщение пользователю + отказ сохранения для текущей операции.  

### 5. Валидация и обработка ошибок

1. **Валидация данных перед сохранением**  
   - До любых операций с файлом остаётся в силе существующая проверка:  
     - `RightsValidator` вызывается на `RoleModel`.  
     - При ошибках в правах (некорректные значения, несогласованные состояния и т.п.) сохранение **не выполняется** ни для Designer, ни для EDT.  
     - Пользователь получает сообщение об ошибке на уровне модели (до файловой операции).  

2. **Ошибки парсинга EDT Rights.xml**  
   - Если при чтении `rightsPath` получен некорректный XML:  
     - Не выполняем частичное восстановление/перезапись;  
     - Показываем пользователю сообщение вроде «Не удалось разобрать Rights.xml (некорректный XML). Изменения через редактор прав не сохранены.»;  
     - Возвращаем ошибку сохранения (VS Code видит, что сохранение не удалось).  

3. **Общие ошибки записи/прав доступа**  
   - Любая ошибка записи (нет прав, файл только для чтения, диск переполнен и т.п.):  
     - Выводим `showErrorMessage` с указанием конкретного пути и базовой причиной;  
     - Возвращаем `saveError`/отклоняем операцию сохранения.  
   - UI остаётся в рабочем состоянии; пользователь может исправить причину и повторить сохранение.  

4. **Конфликты с внешними изменениями**  
   - На первом этапе **не реализуем** отдельный механизм сравнения временных меток/CRC файлов.  
   - Полагаемся на встроенное поведение VS Code для кастомных редакторов и на систему контроля версий (git).  
   - Возможное улучшение на будущее: хранить хэш содержимого, считанного при открытии, и при сохранении сравнивать с текущим состоянием диска, предлагая пользователю merge/overwrite.  

5. **Бэкап EDT Rights.xml**  
   - Явный `.bak` по умолчанию **не создаём**, чтобы не засорять репозиторий и не дублировать возможности git.  
   - При необходимости можно будет ввести настройку расширения (например, `1cMetadataTree.rightsEditor.createBackup`), чтобы перед атомарной заменой копировать `Rights.xml` в `Rights.xml.bak`. Эта возможность не входит в текущий обязательный объём реализации.  

### 6. Резюме поведения при сохранении

1. **Designer (Role.xml)**  
   - Валидация `RoleModel` → сериализация через `RoleXmlSerializer` → атомарная запись в `currentRoleModel.filePath` (Role.xml) с обработкой ошибок.  

2. **EDT (Roles/ИмяРоли.xml)**  
   - Определение формата по `roleModel.metadata.format === ConfigFormat.EDT`.  
   - Вычисление `rightsPath` (`Roles/ИмяРоли/Ext/Rights.xml`) по `RoleModel.filePath`.  
   - Валидация `RoleModel`.  
   - Чтение и парсинг существующего `Rights.xml` (или создание нового DOM при отсутствии файла).  
   - Слияние `RoleModel.rights` с DOM по описанным правилам (обновление/добавление только известных прав, сохранение всех ограничений и шаблонов).  
   - Сериализация DOM обратно в EDT-формат `Rights.xml`.  
   - Атомарная запись в `rightsPath` с обработкой ошибок и без обязательного `.bak`.
